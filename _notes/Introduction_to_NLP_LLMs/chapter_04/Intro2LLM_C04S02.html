Section Goal

This section moves from loading a model to actually using it. 4.2.2. The Pipeline Abstraction#

Once a model and tokenizer are loaded, the pipeline abstraction from Hugging Face provides the simplest path to generating text. It wraps tokenization, model inference, and decoding into a single callable, handling all tensor management internally.

Show code cell source

Hide code cell source

 
from transformers import pipeline
from transformers import AutoModelForCausalLM, AutoTokenizer

# Load the instruction-tuned Phi-3 model.
# `device_map="auto"` places model shards on available GPU/CPU devices automatically.
# `dtype="auto"` selects an efficient precision for the detected hardware.
# `trust_remote_code=False` avoids executing repository-defined custom Python code.
model = AutoModelForCausalLM.from_pretrained(
    "microsoft/Phi-3-mini-4k-instruct",
    device_map="auto",
    dtype="auto",
    trust_remote_code=False,
)

# Load the matching tokenizer so chat templates/special tokens align with the model.
tokenizer = AutoTokenizer.from_pretrained("microsoft/Phi-3-mini-4k-instruct")

# Build a text-generation pipeline that handles tokenization + generation + decoding.
# `temperature=None` suppresses warnings during greedy decoding (`do_sample=False`).
generator = pipeline(
    "text-generation",
    model=model,
    tokenizer=tokenizer,
    temperature=None
)

Device set to use cuda:0

Why set temperature=None in the pipeline?

By default, the Phi-3 model configuration contains a default temperature value. When we run the model with greedy decoding (do_sample=False), Hugging Face will throw a warning because temperature is ignored during greedy search. Setting temperature=None when initializing the pipeline suppresses this warning and keeps our notebook outputs clean. We will dive deeper into how temperature works under the Generation Parameters section below!

The pipeline accepts either raw strings (for base models) or structured message lists (for instruction-tuned models). For instruction-tuned models, the structured format is strongly preferred: it triggers the model’s chat template, which formats the conversation correctly for the model’s fine-tuning setup.

Key Term — Chat Template

A chat template is a model-specific formatting scheme that converts a list of role-tagged messages (system, user, assistant) into a single string with special tokens that the model was trained to recognize. For example, Phi-3 wraps messages in <|user|> / <|end|> / <|assistant|> markers; Llama-3 uses [INST] / [/INST] delimiters. Without the correct template, an instruction-tuned model may produce low-quality or incoherent responses, because the token patterns it learned during fine-tuning are absent. The pipeline API handles template application automatically when you pass both model and tokenizer.

Key Term — Greedy vs. Sampling Decoding

At each generation step, the model produces a probability distribution over its entire vocabulary. How you choose the next token from that distribution is called the decoding strategy:

  • Greedy decoding (do_sample=False): always pick the single most probable token. Deterministic and fast, but can get stuck in repetitive loops.

  • Sampling (do_sample=True): draw a token randomly according to the distribution. Introduces variability — useful for creative tasks, problematic for factual ones.

  • Beam search: maintain the top-k partial sequences at each step and return the globally most probable complete sequence. More expensive but often preferred for translation and summarization where precision matters. Controlled by num_beams in Hugging Face generate().

The parameters below (temperature, top_p, top_k) only apply when do_sample=True.

4.2.4.2. do_sample and temperature#

When do_sample=False, the model uses greedy decoding: it always selects the single most probable next token. This produces deterministic, conservative output.

When do_sample=True, temperature rescales the raw logit scores before converting them to probabilities. Concretely, if the model produces logit vector \(\mathbf{z}\), the probability assigned to token \(i\) is [Goodfellow et al., 2016]:

(4.1)#\[\begin{equation} P(w_i) = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)} \end{equation}\]

where \(T\) is the temperature. When \(T < 1\), the differences between logits are amplified, concentrating probability mass on the highest-scoring tokens. When \(T > 1\), the distribution is flattened toward uniform.

Table 4.2 Temperature Behavior and Outputs#

Temperature Value

Distribution Shape

Core Behavior

temperature = 0.1

Very peaked distribution

Near-deterministic, highly conservative, and focused

temperature = 1.0

Unmodified distribution

Balanced creativity, variety, and natural coherence

temperature = 2.0

Very flat (uniform) distribution

Highly random, chaotic, and often completely incoherent

  • Low Temperature (< 1.0): Sharpens the distribution. It heavily favors high-probability words while suppressing low-probability ones, making the text predictable and steady.

  • High Temperature (> 1.0): Flattens the distribution. It elevates the chances of unexpected or less likely words, leading to more creative, surprising, or erratic outputs.

Note

If you try to pass a temperature value while do_sample=False, the generation framework will throw a warning. This happens because temperature only alters the distribution for sampling—it has no effect on greedy selection. To silence the warning during deterministic runs, explicitly pass temperature=None.

4.2.4.3. top_p (Nucleus Sampling)#

top_p (also called nucleus sampling) restricts sampling to the smallest set of tokens whose cumulative probability meets or exceeds \(p\). Formally, define the nucleus \(\mathcal{V}_p\) at each step as [Holtzman et al., 2019]:

(4.2)#\[\begin{equation} \mathcal{V}_p = \arg\min_{V' \subseteq V} \left\{ V' : \sum_{w \in V'} P(w) \geq p \right\} \end{equation}\]

where tokens in \(V'\) are ranked from highest to lowest probability. Setting top_p=0.9 means: at each step, rank all tokens by probability, take the smallest subset that together accounts for 90% of the probability mass, and sample only from that subset. This prevents the model from sampling extremely low-probability tokens while still allowing meaningful diversity.

Unlike top_k, the nucleus size varies dynamically: when the model is confident (peaked distribution), the nucleus is small; when the model is uncertain, more tokens are included.

4.2.4.4. repetition_penalty#

Penalizes tokens that have already appeared in the generated text, reducing the tendency to repeat phrases. The penalty is applied by dividing the logit of any previously seen token \(w\) by the penalty factor \(\theta\) [Keskar et al., 2019]:

(4.3)#\[\begin{equation} z'_w = \begin{cases} z_w / \theta & \text{if } w \text{ has appeared before} \\ z_w & \text{otherwise} \end{cases} \end{equation}\]

A value of 1.0 applies no penalty (\(\theta = 1\), leaving logits unchanged); values above 1.0 increasingly discourage repetition. Typical values range from 1.1 to 1.3 for instruction-following tasks.

4.2.4.5. top_k#

top_k restricts sampling to the k most probable tokens at each step, zeroing out the probability of all other tokens before sampling. Setting top_k=50 means: at each step, only consider the 50 highest-probability tokens. It is often used together with top_p: for example, top_k=50, top_p=0.9 first restricts to the top 50 candidates and then further restricts to the minimum subset covering 90% of the probability mass.

Choosing Decoding Parameters in Practice

A commonly used starting configuration for instruction-tuned models is do_sample=True, temperature=0.7, top_p=0.9, top_k=50, repetition_penalty=1.1. For deterministic, factual tasks (e.g., information extraction, classification-like prompts), switch to do_sample=False. Adjust temperature first — it has the most noticeable effect — then fine-tune with top_p and top_k if needed.

4.2.5. Quantization: Running Large Models in Limited VRAM#

Full-precision (float32) model weights consume 4 bytes per parameter; half-precision (float16 / bfloat16) reduces this to 2 bytes per parameter. A 7B-parameter model in float16 requires approximately 14 GB of VRAM.

Quantization further reduces memory by representing weights at lower bit-widths:

Table 4.3 VRAM Requirements by Precision for a 7B Parameter Model#

Precision

Bytes/param

VRAM for 7B model

float32

4 bytes

~28 GB

float16 / bfloat16

2 bytes

~14 GB

4-bit (NF4)

0.5 bytes

~4 GB

Key Term — Quantization

Quantization compresses a model’s weight tensors from high-precision floating-point values (float32 or float16) to lower-precision representations such as 8-bit integers (INT8) or 4-bit integers (INT4/NF4). Because each weight occupies fewer bytes in memory, the total VRAM requirement shrinks by 2–8×. The trade-off is a small degradation in output quality, which in practice is often negligible for instruction-following and conversational tasks. The bitsandbytes library provides drop-in quantization compatible with Hugging Face transformers.

The bitsandbytes library, integrated with Hugging Face transformers, enables 4-bit and 8-bit quantization:

Show code cell source

Hide code cell source

 
import torch
from transformers import BitsAndBytesConfig

# Configure 4-bit quantization to reduce VRAM usage during inference.
quantization_config = BitsAndBytesConfig(
    # Store model weights in 4-bit precision (NF4 by default).
    load_in_4bit=True,
    # Use bfloat16 for internal matmul/compute ops.
    bnb_4bit_compute_dtype=torch.bfloat16,
    # Apply nested quantization for additional memory savings.
    bnb_4bit_use_double_quant=True,
)

# Load the model with quantized weights and automatically place layers on available devices.
model = AutoModelForCausalLM.from_pretrained(
    "microsoft/Phi-3-mini-4k-instruct",
    quantization_config=quantization_config,
    device_map="auto",
)

4-bit quantization reduces VRAM usage by roughly 4× with modest quality degradation on most tasks — making it practical to run 7B models on a free Colab T4 (15 GB VRAM) that would otherwise be too large.

When to use quantization

Use 4-bit quantization (load_in_4bit=True) when your GPU has less than 16 GB of VRAM and you are running inference-only workloads (question answering, summarization, extraction). For fine-tuning or tasks requiring high numerical precision, prefer float16/bfloat16 if VRAM allows. The quality difference is usually negligible for instruction-following but may be noticeable for arithmetic-heavy reasoning tasks.

4.2.6. Applying LLMs to Practical NLP Tasks#

Instruction-tuned LLMs can be applied to a wide range of NLP tasks through careful prompt design alone, without any additional training. The key insight is that these models have been fine-tuned to follow natural-language instructions, so the structure and wording of your prompt directly shapes the output.

The three tasks below — question answering, summarization, and structured extraction — illustrate how a single model can serve very different purposes by changing only the system prompt and output format specification.

4.2.6.3. Structured Information Extraction#

Structured extraction asks the model to output data in a machine-readable format (e.g., JSON) rather than natural prose. Two prompt-design practices are critical here:

  1. Enumerate the exact fields you want in the system prompt.

  2. Instruct the model to return only the JSON with no surrounding explanation — otherwise many models will prefix the JSON with a sentence like “Here is the extracted information:”, which breaks downstream parsing.

Use do_sample=False for extraction tasks; stochastic sampling introduces unnecessary variation in field names or formatting.

# Use greedy decoding for deterministic, parser-friendly extraction output.
# `max_new_tokens` caps response length to avoid unnecessary text.
output = generator(extract_messages, max_new_tokens=100, do_sample=False)

# Pipeline returns nested chat output; print only assistant message content.
print(output[0]["generated_text"][-1]["content"])

4.2.7. Exercises and discussion#

  1. Prompt sensitivity Take a simple factual question (e.g., “Explain what gradient descent is”) and run it through the model with three different system prompts: (a) no system prompt, (b) “You are a helpful assistant”, and (c) “You are a professor teaching an undergraduate NLP course.” Compare the responses. How does the system prompt affect length, vocabulary level, and structure?

  2. Temperature sweep Generate five responses to the same creative prompt (e.g., “Write an opening sentence for a mystery novel set in a data center”) at temperatures 0.1, 0.5, 1.0, 1.5, and 2.0. At which temperature do responses become incoherent? At which are they too conservative?

  3. Structured extraction reliability Run the structured extraction prompt above on five different input sentences of your own design. How consistently does the model return valid JSON? What kinds of inputs cause it to fail (e.g., missing fields, ambiguous roles)?
