4.3. LLM Robustness and Model Comparison#

4.3.1. Learning objectives#

By the end of this section, you should be able to:

  • Design a prompt evaluation framework with original and augmented prompt variants.

  • Apply three prompt augmentation strategies: paraphrase, typos/slang, and instruction-style variation.

  • Compute and interpret four quantitative output metrics: response length, type-token ratio, readability score, and n-gram diversity.

  • Analyze robustness degradation across augmentation conditions for two or more LLMs.

  • Critically compare LLMs along axes beyond accuracy: training data philosophy, safety framing, and response style.

Section Goal

This section moves from running a single model on clean prompts to systematically measuring how model quality changes when prompts are noisy, rephrased, or adversarially perturbed. You will build a reusable evaluation framework, apply it to two model pairs across two domains, and learn to interpret the four quantitative metrics that capture different dimensions of output quality. The emphasis is on developing a critical, empirical mindset toward LLM evaluation rather than accepting first-pass outputs at face value.

4.3.2. Why Robustness Evaluation Matters#

Running a model on a clean, well-formed prompt tells you its best-case behavior. In real-world deployments, however, users rarely write perfectly structured prompts: they misspell words, switch between formal and informal registers, phrase the same request five different ways, and occasionally introduce adversarial noise intentionally or accidentally.

Robustness asks a different question than raw quality: does performance degrade gracefully when prompts deviate from the ideal? A model that achieves excellent quality on clean prompts but collapses when a word is misspelled may be unsuitable for consumer-facing applications. The practices in this section (p401_robustness_phi3_stablelm, p402_robustness_gemma_phi3_health) build systematic frameworks for measuring this degradation.

Key Term — Robustness

In machine learning, robustness refers to a model’s ability to maintain performance when its inputs deviate from the training distribution. For LLMs, robustness is evaluated across several axes: linguistic variation (paraphrases, synonyms), surface noise (typos, abbreviations, slang), structural changes (different instruction formats), and adversarial perturbations (deliberately misleading or confusing prompts). A robust model produces consistently useful outputs across all these input variants; a brittle model degrades sharply on even minor deviations.

4.3.3. Prompt Augmentation Strategies#

Given a set of original prompts on a theme (university programs, health and wellness), three augmentation strategies are applied to each prompt:

Key Term — Prompt Augmentation

Prompt augmentation is the process of systematically generating multiple variants of an original prompt to test how sensitive a model’s output is to input form. Unlike data augmentation in supervised learning (where you augment training examples), here you augment evaluation prompts. Each variant preserves the core information need but changes some surface property — vocabulary, word order, register, or noise level. Comparing model outputs across variants reveals which models generalize beyond the exact wording of the prompt.

4.3.3.1. 1. Paraphrase#

The semantic content is preserved but the surface form is changed — synonyms are substituted, sentence structure is reorganized, and phrasing is varied. This tests whether the model’s output quality depends on the specific words used or on the underlying meaning.

Original:   "What career paths are available for a data science graduate?"
Paraphrase: "Which professional routes can someone with a data science degree pursue?"

4.3.3.2. 2. Typos and Slang#

Common typing errors (transpositions, missing letters, phonetic misspellings), abbreviations, and informal language are introduced. This tests robustness to the natural noise present in user-generated text.

Original: "Explain the admission requirements for an MBA program."
Noisy:    "wut r the admissin requirments 4 an MBA prog lol"

4.3.3.3. 3. Instruction-Style Variation#

The same semantic content is expressed with different instructional framing: imperative (“List the…”), interrogative (“What are the…”), and conversational (“Can you tell me about…”) forms. This tests whether the model is sensitive to the surface form of the instruction.

Original:       "Describe the key differences between online and in-person learning."
Imperative:     "List the main differences between online and in-person learning."
Conversational: "Hey, I was wondering if you could explain how online learning differs from going to class in person?"

4.3.4. Quantitative Output Metrics#

Four metrics quantify output quality and generation tendencies across conditions. All four are computed on the raw text of the model’s response; no reference output is required, making them reference-free metrics that can be applied to any generation task.

4.3.4.1. Response Length#

The number of whitespace-delimited tokens in the generated response. This is a simple word-count proxy for token count (the two correlate closely for English text).

Substantial length reductions under noisy prompts indicate the model becomes less informative when input is unclear. Conversely, unexpectedly long responses to simple prompts may indicate verbosity or repetition.

Show code cell source

Hide code cell source

 
def response_length(text):
    # Count words by splitting on whitespace; used as a simple length proxy.
    return len(text.split())

4.3.4.2. Type-Token Ratio (TTR)#

The ratio of unique word types to total word tokens. A higher TTR indicates greater lexical diversity; a lower TTR suggests repetitive output.

(4.4)#\[\begin{equation} \text{TTR} = \frac{|\text{unique words}|}{|\text{total words}|} \end{equation}\]

Limitation — Length Bias

TTR decreases as text length increases, even when lexical richness stays constant, because longer texts are more likely to repeat common function words. This means raw TTR scores are not directly comparable across responses of very different lengths. When comparing models that produce responses of substantially different length, prefer MATTR (Moving-Average Type-Token Ratio), which computes TTR over a sliding fixed-length window and averages across windows, making it length-invariant. For the short responses typical of LLM evaluations (50–300 words), raw TTR is generally sufficient.

Show code cell source

Hide code cell source

 
def type_token_ratio(text):
    # Normalize to lowercase and split on whitespace to get word tokens.
    words = text.lower().split()
    # TTR = unique tokens / total tokens; return 0 for empty input.
    return len(set(words)) / len(words) if words else 0

Example:

Hide code cell source

 
# Demonstrate lexical diversity using Type-Token Ratio (TTR)

4.3.4.3. Readability (Flesch Reading Ease)#

The Flesch Reading Ease score estimates how difficult a text is to read, based on average sentence length and average syllable count per word:

(4.5)#\[\begin{equation} \text{FRE} = 206.835 - 1.015 \left(\frac{\text{total words}}{\text{total sentences}}\right) - 84.6 \left(\frac{\text{total syllables}}{\text{total words}}\right) \end{equation}\]

Scores range from 0 (very difficult — dense academic prose) to 100 (very easy — simple conversational text). Typical ranges:

Score range

Approximate reading level

90–100

Very easy (5th grade)

60–70

Standard (8th–9th grade)

30–50

Difficult (college level)

0–30

Very difficult (professional/technical)

In robustness evaluation, a drop in readability score under noisy prompts can indicate that the model is producing longer, more convoluted sentences as it struggles to interpret unclear input.

Hide code cell source

 
def readability_score(text):
    # Calculate Flesch Reading Ease score using textstat library.
    # Scores range from 0 (very difficult) to 100 (very easy).
    return textstat.flesch_reading_ease(text)

4.3.4.4. N-gram Diversity#

The ratio of unique n-grams to total n-gram occurrences, measuring how much the model repeats short phrases rather than generating varied text. Defined for n-gram order \(n\) as:

(4.6)#\[\begin{equation} \text{Diversity}_n = \frac{|\{g : g \text{ is a unique } n\text{-gram in } w\}|}{|\text{total } n\text{-grams in } w|} \end{equation}\]

A score of 1.0 means every n-gram in the response is unique (maximum diversity); a score near 0 indicates near-total repetition. Bigrams (\(n=2\)) are most commonly used; trigrams (\(n=3\)) are more sensitive to phrase-level repetition.

Relation to Self-BLEU

N-gram diversity is the complement of self-BLEU, a metric used in text generation research to measure how similar a set of generated samples are to each other. High self-BLEU (low diversity) signals mode collapse — the model produces nearly identical responses regardless of minor prompt variation. When applied to a single response rather than a set, n-gram diversity captures intra-response repetition specifically.

Show code cell source

Hide code cell source

 
def ngram_diversity(text, n=2):
    # Tokenize on whitespace; this keeps punctuation attached to tokens.
    tokens = text.split()
    
    # Build all contiguous n-grams of size `n` from the token sequence.
    all_ngrams = list(ngrams(tokens, n))
    
    # Convert to a set to keep only unique n-grams.
    unique_ngrams = set(all_ngrams)
    
    # Diversity = unique n-grams / total n-grams; return 0 if text is too short.
    return len(unique_ngrams) / len(all_ngrams) if all_ngrams else 0

Example:

Show code cell source

Hide code cell source

 
# Calculate N-Gram Diversity (using Bigrams, n=2)
print("--- N-GRAM DIVERSITY (n=2) ---")
# Simple text repeats phrases like "the mouse" and "chased the"
print(f"Simple Text Diversity:  {ngram_diversity(simple_text, n=2):.2f} (Contains repetitive phrases)")

4.3.5. Exercises and Discussion#

  1. Metric selection
    Of the four output metrics introduced in this section (length, TTR, readability, n-gram diversity), which is most likely to detect a model that responds to noisy prompts by repeating the same phrase? Which would you use to detect a model that produces very short, unhelpful responses under adversarial conditions?

  2. Adversarial prompt design
    Design three adversarial prompts in a domain of your choice (e.g., finance, legal, education) that represent vague framing, misleading context, and urgency injection. Run them through two models and compare the responses qualitatively.

  3. Safety vs. helpfulness trade-off
    Models trained with RLHF tend to add safety hedges (e.g., “consult a professional”) more frequently in sensitive domains. Describe a scenario where excessive hedging is a failure mode rather than a feature. How would you measure this trade-off quantitatively?
