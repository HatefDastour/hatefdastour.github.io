4.4. Tokens and Tokenization#

4.4.1. Learning objectives#

By the end of this section, you should be able to:

  • Explain what a token is and why text must be tokenized before a model can process it.

  • Describe why character-level and word-level tokenization fail at scale and how subword tokenization addresses these failures.

  • Explain the core algorithmic difference between WordPiece (BERT), Byte-Pair Encoding (GPT-2), and SentencePiece (Flan-T5).

  • Inspect raw token ID sequences and decode them back into text to understand exactly what the model receives as input.

  • Identify special tokens ([CLS], [SEP], <|assistant|>, <pad>) and explain the role each plays.

Section Goal

Before a language model can process text, the text must be converted into integers. This section explains the entire tokenization pipeline: why it exists, what can go wrong with naive approaches, and how modern subword algorithms solve the coverage-efficiency trade-off. By the end, you will be able to tokenize any string with BERT, GPT-2, or Flan-T5, inspect the resulting IDs, decode them back to text, and identify the special tokens that appear automatically.

4.4.2. What is a Token?#

A token is the fundamental unit of text that a language model processes. Every model has a fixed vocabulary of tokens — a lookup table mapping integer indices (token IDs) to pieces of text. The model never sees characters or words directly: it sees sequences of integers, each representing a token from this vocabulary.

Key Term — Vocabulary and Token ID

A tokenizer’s vocabulary is the complete set of text pieces it knows, each assigned a unique integer called a token ID. For example, BERT has a vocabulary of ~30,522 entries; GPT-2 has 50,257. When the tokenizer processes a string, it returns a sequence of integers — the token ID sequence — which is the only representation the model ever works with. All subsequent computation (embedding lookup, transformer layers, output projection) operates on these integer sequences.

Tokens are not words. Depending on the tokenization algorithm, a token may be:

  • A whole common word: "hello" → ID 15339

  • A word fragment (subword): "##ing", "Ġtrans", "▁forma", "tion"

  • A single character: "z", "5", "!"

  • A special symbol: [CLS], </s>, <|user|>

  • A byte (in byte-level tokenizers): Ġ0x41

The journey from text to model input always passes through tokenization:

"Mizzou is a top research university."
              ↓  tokenizer
    Token IDs: [1, 59172, 338, 263, 2246, 5925, 3014, 29889, 2]
              ↓  embedding lookup
    Token vectors (one per ID) → Transformer layers → output

4.4.3. Why Not Use Words or Characters?#

Word-level tokenization splits on whitespace and punctuation, producing one token per word. This approach has two critical problems:

  1. Vocabulary explosion: English has millions of words including inflected forms, compounds, and proper nouns. A vocabulary large enough to cover all words would require billions of embeddings, making the model impractical.

  2. Out-of-vocabulary (OOV) tokens: Any word not seen during training gets mapped to a special [UNK] token, collapsing all unknown words into a single meaningless representation. New terms, technical vocabulary, and misspellings are systematically lost.

Character-level tokenization avoids the OOV problem — any string is representable as a character sequence — but introduces a different problem: sequences become extremely long. A 512-word passage becomes roughly 2,500 characters, making it difficult for the model to learn long-range dependencies across the sequence.

Subword tokenization strikes a balance: common words are represented as single tokens (short sequences), while rare or unknown words are split into familiar subword pieces. The total vocabulary size can be kept to 30,000–100,000 tokens while maintaining full coverage of any input.

Key Term — Subword Tokenization

Subword tokenization is a family of algorithms that segment text into pieces that are larger than single characters but smaller than whole words. The key insight is that many words share meaningful fragments — “play”, “plays”, “played”, “player” all contain “play”. By representing these as a root subword plus affixes, a model can generalize across word forms. Unlike word-level tokenization, subword schemes guarantee full coverage: any unseen word can be decomposed into known pieces (or individual bytes), eliminating the out-of-vocabulary problem.

4.4.4. Subword Tokenization Algorithms#

Three algorithms dominate modern NLP. All three start from the same observation — a fixed vocabulary should balance frequency (common sequences get their own token) against coverage (rare sequences can always be decomposed) — but they differ in how they build that vocabulary and how they apply it at inference time.

4.4.4.1. WordPiece (BERT — bert-base-uncased)#

WordPiece builds its vocabulary by iteratively selecting merges that maximise the likelihood of the training corpus under a unigram language model. At each step it scores every candidate pair \((A, B)\) and picks the merge that gives the largest gain:

(4.7)#\[\begin{equation} \text{score}(A, B) = \frac{\text{freq}(AB)}{\text{freq}(A) \times \text{freq}(B)} \end{equation}\]

This ratio rewards pairs that co-occur far more often than chance — essentially a pointwise mutual information criterion. The algorithm operates on Unicode characters (not bytes), lowercases all text, and prefixes continuation subwords (those that are not word-starts) with ##.

Input: "Tokenization is fascinating."
BERT:  ["token", "##ization", "is", "fascinating", "."]

Vocabulary size: ~30,522. Any character not in the vocabulary is mapped to [UNK], so emoji, rare Unicode symbols, and many non-Latin scripts are silently collapsed to a single unknown token.

4.4.4.2. Byte-Pair Encoding (BPE — GPT-2, RoBERTa)#

BPE initialises the vocabulary with all individual characters (or bytes), then iteratively merges the most frequent adjacent pair across the entire corpus until a target vocabulary size is reached:

(4.8)#\[\begin{equation} \text{merge} = \underset{(A,\,B)}{\arg\max}\; \text{freq}(AB) \end{equation}\]

GPT-2 applies BPE at the byte level: the initial vocabulary consists of all 256 possible byte values, so any Unicode string can be encoded without ever emitting [UNK]. Word boundaries are marked with a space prefix shown as Ġ.

Input: "Tokenization is fascinating."
GPT-2: ["Token", "ization", "Ġis", "Ġfascinating", "."]

Vocabulary size: 50,257. The byte-level guarantee means even emoji and mathematical symbols are representable — though they may expand into several byte tokens.

4.4.4.3. SentencePiece with Unigram LM (Flan-T5 — google/flan-t5-small)#

SentencePiece trains a unigram language model directly on raw text without any whitespace pre-tokenisation. It starts with a large candidate vocabulary and prunes it by repeatedly removing the token whose removal causes the smallest drop in corpus log-likelihood:

(4.9)#\[\begin{equation} \mathcal{L}(\mathcal{V}) = \sum_{x \in \mathcal{D}} \log P(x \mid \mathcal{V}), \qquad P(x \mid \mathcal{V}) = \max_{\mathbf{s} \in S(x)} \prod_{i} p(s_i) \end{equation}\]

where \(S(x)\) is the set of all possible segmentations of sentence \(x\) and \(p(s_i)\) is the unigram probability of subword \(s_i\). Word-starting subwords are prefixed with ▁.

Input: "Tokenization is fascinating."
Flan-T5: ["▁Token", "ization", "▁is", "▁fascin", "ating", "."]

Vocabulary size: 32,100. Because the model is probabilistic, the same string can in principle be segmented in multiple ways; at inference time the most probable segmentation (via the Viterbi algorithm) is used.

Note — Special tokens and the token ID sequence

Special tokens are invisible when you call .tokenize() because that method returns only the surface pieces. They do appear in the integer sequence returned by .encode(). This means the model receives more tokens than the raw text contains — always account for special tokens when estimating prompt length against a context window limit.

4.4.6. Inspecting Token IDs#

Three AutoTokenizer methods cover the full round-trip between text and integers:

  • .tokenize(text) — splits text into subword strings; special tokens are not added.

  • .encode(text) — converts text to a list of integer IDs; special tokens are added automatically.

  • .decode(ids) — converts a list of integer IDs back to a human-readable string, including any special tokens.

The example below demonstrates all three on a sentence about the University of Missouri:

Show code cell source

Hide code cell source

 
from transformers import AutoTokenizer

# BERT tokenizer
bert_tok = AutoTokenizer.from_pretrained("bert-base-uncased")
text = "The University of Missouri was founded in 1839."

tokens = bert_tok.tokenize(text)
ids    = bert_tok.encode(text)

print("Tokens :", tokens)
print("IDs    :", ids)
print("Decoded:", bert_tok.decode(ids))

Tokens : ['the', 'university', 'of', 'missouri', 'was', 'founded', 'in', '1839', '.']
IDs    : [101, 1996, 2118, 1997, 5284, 2001, 2631, 1999, 10011, 1012, 102]
Decoded: [CLS] the university of missouri was founded in 1839. [SEP]

Tokens : ['the', 'university', 'of', 'missouri', 'was', 'founded', 'in', '1839', '.']
IDs    : [101, 1996, 2118, 1997, 4585, 2001, 2631, 1999, 13423, 1012, 102]
Decoded: [CLS] the university of missouri was founded in 1839. [SEP]

Notice that IDs 101 and 102 correspond to [CLS] and [SEP] — added automatically by BERT’s tokenizer even though .tokenize() did not show them. The decoded output reveals these structural tokens: 9 words of text became an 11-token sequence.

4.4.7. Token Count and Context Window Implications#

Every model has a maximum context length — the maximum number of tokens it can process in a single forward pass. Tokens consumed by the prompt reduce the tokens available for the model’s response. Some examples from models covered in this course:

Max context (tokens)

Key Term — Fertility

Fertility (also called token fertility) measures how many tokens a tokenizer produces per word on a given text:

(4.10)#\[\begin{equation} \text{fertility} = \frac{\text{number of tokens}}{\text{number of whitespace-delimited words}} \end{equation}\]

A fertility of 1.0 means every word maps to exactly one token — ideal but rare. Values above 1.5 on standard English text usually signal either a mismatch between the tokenizer’s training domain and the input, or the presence of rare vocabulary (proper nouns, code, non-Latin scripts). Higher fertility means more tokens, which fills the context window faster and increases API cost.

Because different tokenizers are trained on different corpora with different vocabulary sizes, the same sentence can produce very different token counts. The cells below load all three tokenizers and compare them on a single sentence:

Show code cell source

Hide code cell source

 
from transformers import AutoTokenizer

bert_tok = AutoTokenizer.from_pretrained("bert-base-uncased")
gpt2_tok = AutoTokenizer.from_pretrained("gpt2")
t5_tok   = AutoTokenizer.from_pretrained("google/flan-t5-small")

Show code cell source

Hide code cell source

 
text = "The University of Missouri offers 300+ degree programs across 19 colleges."

for name, tok in [("BERT", bert_tok), ("GPT-2", gpt2_tok), ("Flan-T5", t5_tok)]:
    count = len(tok.encode(text))
    words = len(text.split())
    fertility = round(count / words, 2)
    print(f"{name:10s}: {count:3d} tokens  ({words} words → fertility {fertility})")

BERT      :  15 tokens  (11 words → fertility 1.36)
GPT-2     :  13 tokens  (11 words → fertility 1.18)
Flan-T5   :  14 tokens  (11 words → fertility 1.27)

The same sentence can require 6 tokens under one tokenizer and 14 under another. For API providers (OpenAI, Anthropic, Google) that charge per token, this difference directly affects both context window capacity and cost.

4.4.8. Exercises and discussion#

  1. OOV analysis with WordPiece Feed BERT’s tokenizer five sentences containing proper nouns, technical jargon, and informal abbreviations. For each sentence, identify which tokens were split into subwords and which were replaced with [UNK]. What pattern determines whether a word is split vs. unknown?

  2. Byte-level vs. character-level comparison Take a sentence containing an emoji and two non-Latin characters (e.g., Chinese or Arabic). Compare how BERT’s WordPiece tokenizer and GPT-2’s byte-level BPE handle it. Count the resulting tokens and decode the output. What does this reveal about the two approaches to unknown characters?

  3. Token efficiency across tokenizers Choose five sentences covering different text types: formal English, informal slang, mathematical notation, Python code, and a sentence in a non-English language. Tokenize each with BERT, GPT-2, and Flan-T5 and compare token counts. Which tokenizer is most efficient for each text type? Can you explain why based on their vocabulary construction approaches?

  4. Fertility in practice Compute the fertility of each tokenizer on the five sentences from Exercise 3. For the sentence with the highest fertility, inspect which specific words or characters are being split the most. Does this align with the algorithmic properties described in this section?
