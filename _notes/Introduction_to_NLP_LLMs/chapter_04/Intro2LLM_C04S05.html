4.5. Token Embeddings and the Text-to-Numbers Pipeline#

4.5.1. Learning Objectives#

By the end of this section, you should be able to:

  • Describe the four-layer pipeline from raw text to sentence embedding: tokenization, static embedding lookup, transformer contextualization, and pooling.

  • Distinguish between static token embeddings (one vector per token type) and contextualized embeddings (one vector per token instance).

  • Obtain token-level contextualized embeddings from a transformer encoder and interpret the shape of the output tensor.

  • Produce sentence-level embeddings using Sentence Transformers and apply them to compute semantic similarity.

  • Explain why semantic similarity is a more useful downstream signal than raw cosine distance between averaged word vectors.

Section Goal

This section traces the complete path from a raw text string to a single numerical vector that represents its meaning. Understanding this pipeline is essential for any application that goes beyond text generation: semantic search, clustering, recommendation, and retrieval-augmented generation all rely on sentence embeddings as their foundation. By the end, you will have computed contextualized embeddings with DeBERTa and sentence embeddings with Sentence Transformers, and will understand when each is appropriate.

4.5.2. The Four-Layer Pipeline#

Everything in this course — from the first bag-of-words model to the large language models of the previous sections — requires one foundational transformation: converting human-readable text into numbers. This transformation proceeds through four distinct layers:

Layer 1: Tokenization
  "Mizzou is a top research university."
   ↓
  Token IDs: [1, 59172, 338, 263, 2246, 5925, 3014, 29889, 2]
  (The ONLY thing the model ever sees)

Layer 2: Static Embedding Lookup
  "Mizzou"     → [0.12, -0.34,  0.87, ...]  (768-dim vector)
  "research"   → [0.44, -0.11,  0.63, ...]
  One vector per token type, always the same regardless of context.

Layer 3: Transformer Layers (Contextualization)
  "bank" in "river bank"     → [0.83, -0.21,  0.94, ...]
  "bank" in "bank account"   → [0.61,  0.44, -0.37, ...]
  Now each vector reflects the full surrounding context.

Layer 4: Pooling (Sentence-Level)
  [ 0.21, -0.05,  0.74, ...]
  ONE vector for the whole sentence.
  Used for search, clustering, classification, recommendations.

Understanding each layer separately — and how they compose — is essential for choosing the right tool for a given task.

4.5.3. Layer 2: Static Embedding Lookup#

The first trainable component inside a transformer model is an embedding matrix \(E \in \mathbb{R}^{V \times d}\), where \(V\) is the vocabulary size and \(d\) is the embedding dimension. Given a token ID sequence \([t_1, t_2, \ldots, t_n]\), the embedding layer performs a simple row lookup:

(4.11)#\[\begin{equation} \mathbf{e}_i = E[t_i] \in \mathbb{R}^d \end{equation}\]

Each token ID \(t_i\) selects the corresponding row from \(E\), producing a \(d\)-dimensional vector \(\mathbf{e}_i\). The full input to the transformer is the matrix \([\mathbf{e}_1, \mathbf{e}_2, \ldots, \mathbf{e}_n] \in \mathbb{R}^{n \times d}\).

These are static embeddings: the vector for token bank is identical regardless of whether it appears in a financial context or a riverbank context. The embedding matrix is initialized randomly and updated during pre-training; after training, it captures distributional semantics — tokens that appear in similar contexts have similar vectors.

Key Term — Distributional Semantics

Distributional semantics is the principle that words with similar meanings tend to appear in similar contexts: “You shall know a word by the company it keeps” (J.R. Firth, 1957). This principle underlies all word embedding methods — Word2Vec, GloVe, and the embedding matrices inside transformers. If “doctor” and “physician” consistently appear near words like “hospital”, “patient”, and “diagnosis”, their learned vectors will be close in the embedding space.

4.5.4. Layer 3: Contextualized Embeddings via Transformer Layers#

The transformer stack transforms the sequence of static token embeddings into contextualized embeddings — representations that encode not just the identity of each token, but its meaning given all other tokens in the sequence.

Each transformer layer applies multi-head self-attention. For a single attention head, the computation for query matrix \(Q\), key matrix \(K\), and value matrix \(V\) is:

(4.12)#\[\begin{equation} \text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right)V \end{equation}\]

where \(d_k\) is the key/query dimension (used to prevent the dot products from growing too large). The output of the final transformer layer is a sequence of contextualized vectors \([\mathbf{h}_1, \mathbf{h}_2, \ldots, \mathbf{h}_n]\), one per input token.

Key Term — Self-Attention

Self-attention allows each token to incorporate information from every other token in the sequence (or, in causal/decoder models, every preceding token). For each token, the model computes a weighted combination of all other tokens’ value vectors, where the weights are determined by the query–key dot product: how relevant is each other token to the current one? After applying a softmax, these weights form an attention distribution. Multi-head self-attention runs several such computations in parallel — each “head” may focus on different linguistic relationships — and their outputs are concatenated and projected. This is what gives transformer models their ability to capture both local and long-range dependencies.

Why DeBERTa?

DeBERTa (Decoding-Enhanced BERT with Disentangled Attention) is a strong encoder model from Microsoft that improves on BERT and RoBERTa by disentangling the positional and content components of attention. We use it here because its last_hidden_state outputs serve as clean examples of contextualized token embeddings, and because it is compact enough to run on a free Colab T4 GPU without quantization.

from transformers import AutoTokenizer, AutoModel
import torch

model_name = "microsoft/deberta-v3-base"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModel.from_pretrained(model_name)

text = "Mizzou is a top research university."
# Tokenize text and return PyTorch tensors
inputs = tokenizer(text, return_tensors="pt")

# Disable gradient computation for inference (saves memory)
with torch.no_grad():
    outputs = model(**inputs)

# Extract the last hidden state: one contextualized vector per token
# Shape: (batch_size=1, sequence_length=9, hidden_size=768)
# The 9 tokens include [CLS], "Mizzou", "is", "a", "top", "research", "university", ".", [SEP]
hidden_states = outputs.last_hidden_state
print(f"Output shape: {hidden_states.shape}")

The output tensor has shape (batch_size, sequence_length, hidden_size). For DeBERTa-base with a 10-token input, this is (1, 10, 768): one batch, ten token vectors each of dimension 768.

Key observations:

  • The same word "bank" produces two different 768-dimensional vectors depending on its context — the defining property of contextualized representations.

  • The output is one vector per token position, not one per word type. If "university" is split into two subwords by the tokenizer, there are two corresponding vectors.

  • The [CLS] token’s vector (position 0) is often used as a summary representation for classification tasks, though it was not explicitly trained for that purpose in all models.

4.5.5. Layer 4: Pooling to Sentence Embeddings#

For tasks like semantic search, clustering, or sentence similarity, we need a single fixed-length vector representing the entire sentence. Pooling aggregates the sequence of token vectors \([\mathbf{h}_1, \ldots, \mathbf{h}_n]\) into one vector \(\mathbf{s} \in \mathbb{R}^d\). Common strategies:

Strategy

Formula

Notes

CLS pooling

\(\mathbf{s} = \mathbf{h}_1\)

Simple; uses the [CLS] token vector

Mean pooling

\(\mathbf{s} = \frac{1}{n}\sum_{i=1}^{n} m_i \mathbf{h}_i\)

\(m_i=1\) for real tokens, 0 for padding; generally more stable

Max pooling

\(\mathbf{s}_j = \max_i \, h_{ij}\)

Element-wise max across all positions

Sentence Transformers (sentence-transformers library) provide models explicitly fine-tuned for sentence-level tasks using contrastive learning objectives, producing far better similarity estimates than naive pooling of general-purpose token embeddings.

Key Term — Contrastive Learning and Sentence Transformers

Contrastive learning is a training objective that pulls similar examples together and pushes dissimilar examples apart in embedding space. A Sentence Transformer is fine-tuned on pairs of semantically equivalent sentences (e.g., paraphrase corpora or natural language inference data) with a loss that minimizes the distance between equivalent sentence vectors and maximizes the distance between unrelated ones. The result is an embedding space where cosine similarity directly reflects semantic similarity — “The cat sat on the mat” and “A feline rested on a rug” will be nearby, while “The cat sat” and “The stock market crashed” will be far apart.

Hide code cell source

 
from sentence_transformers import SentenceTransformer

# Load a compact Sentence Transformer model tuned for sentence-level semantics.
model = SentenceTransformer("all-MiniLM-L6-v2")

# Example inputs:
# - Sentences 1 and 2 are semantically similar (paraphrases).
# - Sentence 3 is less similar, so cosine scores should be lower.
sentences = [
    "The student enrolled in a machine learning course.",
    "A learner signed up for an AI class.",
    "The professor submitted the final exam grades.",
]

# Convert each sentence to a dense embedding vector.
# Output shape: (number_of_sentences, embedding_dimension)
embeddings = model.encode(sentences)

print(f"Embedding shape: {embeddings.shape}")  # (3, 384)

Embedding shape: (3, 384)

4.5.6. Semantic Similarity with Sentence Embeddings#

Given sentence embeddings, cosine similarity measures the angle between two vectors, independent of their magnitude:

(4.13)#\[\begin{equation} \text{cosine\_similarity}(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \cdot \|\mathbf{v}\|} \end{equation}\]

Values range from \(-1\) (opposite directions) to \(+1\) (identical directions), with \(0\) indicating orthogonality. For unit-normalized embeddings, cosine similarity equals the dot product \(\mathbf{u} \cdot \mathbf{v}\).

Sentence Transformer embeddings capture semantic similarity rather than surface-form similarity. The pair “The student enrolled in a machine learning course” and “A learner signed up for an AI class” should achieve high cosine similarity despite sharing almost no words, because they describe the same event.

from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

# Compute pairwise cosine similarity between all sentence embeddings.
# The diagonal entries are 1.0 because each sentence is compared with itself.
sims = cosine_similarity(embeddings)

# Print only unique sentence pairs once (i < j) to avoid duplicate comparisons.
for i in range(len(sentences)):
    for j in range(i + 1, len(sentences)):
        print(f"  Sim({i+1},{j+1}) = {sims[i, j]:.4f}")
        print(f"    '{sentences[i][:60]}'")
        print(f"    '{sentences[j][:60]}'")

The complete text-to-numbers pipeline can now be summarized:

Input

Output

Transformer layers

Token vector matrix

Each layer connects to topics covered earlier in the course:

  • Labs 2.01–2.03 (BoW, TF-IDF): Bag-of-words representations bypass the embedding layer entirely, using sparse count vectors instead.

  • Lab 2.04 (Word2Vec): Pre-trained static token embeddings used as fixed features — Layer 2 only, no contextualization.

  • Lab 3.05 (BERT fine-tuning): Contextualized representations from a transformer encoder, fine-tuned end-to-end — Layers 2 and 3.

  • This section: The complete pipeline, from tokenization through pooling to sentence-level semantics — all four layers.

4.5.8. Exercises and Discussion#

  1. Shape interpretation Load DeBERTa and run it on a sentence of your choice. Print the shape of outputs.last_hidden_state. Verify that the second dimension (sequence length) equals the number of tokens produced by the tokenizer for that input, including any special tokens such as [CLS] and [SEP].

  2. Context sensitivity demonstration Create two sentences where the same word appears in different semantic contexts (e.g., “I went to the river bank” and “I deposited money at the bank”). Extract the contextualized embedding for the ambiguous word from each sentence and compute the cosine similarity between the two vectors. Is the similarity high (word identity dominates) or low (context dominates)?

  3. CLS vs. mean pooling comparison Using a Sentence Transformer or a BERT model, compare CLS pooling versus mean pooling for five sentence pairs. Does the choice of pooling strategy change the ranking of similarity scores? For which types of sentences is the difference largest?

  4. Static vs. contextualized embeddings Retrieve the static embedding for a polysemous word (e.g., "plant") directly from the model’s embedding matrix (model.embeddings.word_embeddings). Then obtain two contextualized embeddings for the same word in different sentences. Compare all three vectors. What does this reveal about the information added by the transformer layers?
