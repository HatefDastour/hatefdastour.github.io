5.3.4. Text Classification with Leveraging Embeddings#

5.3.4.1. Learning objectives#

By the end of this section, you should be able to:

  • Apply a pre-trained classifier (RoBERTa fine-tuned for sentiment) to a new dataset without additional training.

  • Generate sentence embeddings using Sentence Transformers and explain what makes them suitable for classification.

  • Train a Logistic Regression classifier on top of sentence embeddings and explain why this two-stage approach is efficient.

  • Implement prototype-based (centroid) classification using cosine similarity and explain why it works geometrically.

  • Implement zero-shot embedding classification by encoding natural-language class descriptions and comparing them to document embeddings.

  • Use a generative model (Flan-T5) for text classification via prompted text generation.

  • Compare all six approaches on accuracy, data requirements, compute cost, and latency.

5.3.4.3. Approach 1: Pre-Trained Sentiment Classifier (RoBERTa)#

The simplest approach: use a RoBERTa model already fine-tuned for sentiment analysis. No additional training is required — we apply it directly to our target dataset.

from transformers import pipeline

sentiment_pipeline = pipeline(
    "text-classification",
    model="cardiffnlp/twitter-roberta-base-sentiment-latest",
)

# Apply to Rotten Tomatoes reviews
results = sentiment_pipeline(test_reviews, truncation=True)
y_pred = [1 if r["label"] == "positive" else 0 for r in results]

Why this works: The model was trained on Twitter data (short, informal text) that shares properties with short critic reviews — colloquial language, opinions expressed in one or two sentences.

Limitation: Domain shift. A model fine-tuned on Twitter may not perfectly reflect the vocabulary and conventions of professional film criticism. Typical accuracy on Rotten Tomatoes: ~78–82%.

5.3.4.4. Approach 2: Sentence Embeddings + Logistic Regression#

Sentence Transformers produce high-quality fixed-length embeddings explicitly trained to capture semantic similarity. The all-mpnet-base-v2 model maps any sentence to a 768-dimensional vector:

5.3.4.5. Background: Text Classification and Semantic Embeddings#

Text classification assigns a predefined category label to a piece of text. Formally, given a document \(x\) and a label set \(\mathcal{Y} = \{y_1, \ldots, y_k\}\), a classifier applies a mapping:

\[f: x \mapsto y^* \in \mathcal{Y}\]

Sentiment analysis — predicting positive vs. negative — is the canonical binary case (\(k = 2\)).

5.3.4.5.1. What Makes a Good Text Representation?#

Raw text cannot be fed directly into classifiers. A good representation must satisfy two properties:

  1. Semantic similarity → geometric proximity. Documents with similar meaning should map to nearby vectors.

  2. Fixed dimensionality. The vector length must be constant regardless of input length.

Sentence Transformers satisfy both. They are encoder models fine-tuned with a contrastive objective: similar sentence pairs are pulled closer in embedding space, while dissimilar pairs are pushed apart.

Definition — Sentence Embedding

A sentence embedding is a fixed-length dense vector \(\mathbf{e} \in \mathbb{R}^d\) produced by a model trained so that semantically similar sentences have embeddings with high cosine similarity. The all-mpnet-base-v2 model produces \(d = 768\)-dimensional vectors.

5.3.4.5.2. Cosine Similarity#

Embedding-based approaches in this section use cosine similarity to compare vectors. For \(\mathbf{a}, \mathbf{b} \in \mathbb{R}^d\):

\[\text{sim}(\mathbf{a}, \mathbf{b}) = \frac{\mathbf{a} \cdot \mathbf{b}}{\|\mathbf{a}\| \, \|\mathbf{b}\|}\]

Values range from \(-1\) (opposite directions) to \(+1\) (identical directions), with \(0\) indicating orthogonality. Cosine similarity is preferred over Euclidean distance for embeddings because it is invariant to vector magnitude — only the direction (semantic content) matters.

5.3.4.6. Approach 2: Sentence Embeddings + Logistic Regression#

The Logistic Regression classifier learns a linear decision boundary in the 768-dimensional embedding space. Given an embedding \(\mathbf{e} \in \mathbb{R}^{768}\), the predicted probability of the positive class is:

\[P(y=1 \mid \mathbf{e}) = \sigma(\mathbf{w}^\top \mathbf{e} + b) = \frac{1}{1 + e^{-(\mathbf{w}^\top \mathbf{e} + b)}}\]

where \(\mathbf{w} \in \mathbb{R}^{768}\) is a weight vector learned during training and \(b\) is the bias term. The document is classified as positive if \(P(y=1 \mid \mathbf{e}) > 0.5\).

from sentence_transformers import SentenceTransformer
from sklearn.linear_model import LogisticRegression

# Step 1: Generate embeddings (run once)
st_model = SentenceTransformer("all-mpnet-base-v2")
X_train_emb = st_model.encode(train_reviews, show_progress_bar=True)
X_test_emb  = st_model.encode(test_reviews,  show_progress_bar=True)
# Shape: (n_samples, 768)

# Step 2: Train a fast Logistic Regression on top
lr = LogisticRegression(max_iter=1000, C=1.0)
lr.fit(X_train_emb, y_train)
y_pred = lr.predict(X_test_emb)

This two-stage approach is efficient because:

  • The expensive embedding step (running all-mpnet-base-v2) is done once per dataset.

  • Logistic Regression training is instantaneous (milliseconds for 8,530 samples).

  • Inference is very fast: a single matrix-vector multiplication per document.

Typical accuracy: ~85–87%. The LR decision boundary is linear in 768-dimensional embedding space — but this space is geometrically structured so that a hyperplane can effectively separate most sentiment classes.

5.3.4.7. Approach 3: Prototype / Centroid Classification#

Instead of training a classifier on labeled embeddings, we compute the centroid (average embedding) of all training documents in each class and classify new documents by proximity:

Definition — Centroid / Prototype

The centroid of class \(c\) is the mean embedding of all training documents in that class:

\[\boldsymbol{\mu}_c = \frac{1}{|\mathcal{D}_c|} \sum_{i \in \mathcal{D}_c} \mathbf{e}_i\]

Classification assigns a new document \(\mathbf{e}\) to the nearest centroid by cosine similarity:

\[\hat{y} = \arg\max_{c} \; \text{sim}(\mathbf{e},\, \boldsymbol{\mu}_c)\]
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

# Compute centroids
for label in [0, 1]:
    mask = np.array(y_train) == label
    centroids[label] = X_train_emb[mask].mean(axis=0)

# Classify by cosine similarity to centroids
def classify_centroid(embedding):
    sims = {label: cosine_similarity([embedding], [centroid])[0, 0]
            for label, centroid in centroids.items()}
    return max(sims, key=sims.get)

Why this works geometrically:

Embedding space (conceptualized in 2D):

  ● ● ●  ← Positive reviews cluster together
      ●

  ■ ■    ← Negative reviews cluster separately
    ■ ■

  Centroid ⊕ = average of all ● positions
  Centroid ⊗ = average of all ■ positions

  New document ★: compute cosine similarity to ⊕ and ⊗,
                  assign to nearer centroid

Sentence Transformer embeddings are trained with contrastive objectives so that semantically similar documents cluster together. This geometric structure makes centroid-based classification competitive despite requiring no classifier training.

Typical accuracy: ~83–85% — slightly below Logistic Regression because no decision boundary optimization occurs.

5.3.4.8. Approach 4: Zero-Shot Embedding Classification#

The most striking approach: classify documents without any labeled training examples by encoding natural-language descriptions of the classes:

# Define class descriptions in natural language
label_descriptions = {
    0: "A negative movie review expressing disappointment or criticism.",
    1: "A positive movie review expressing praise or enthusiasm.",
}

# Encode descriptions and documents into the same embedding space
desc_embeddings = {
    label: st_model.encode(desc)
    for label, desc in label_descriptions.items()
}

def classify_zero_shot(doc_embedding):
    sims = {
        label: cosine_similarity([doc_embedding], [desc_emb])[0, 0]
        for label, desc_emb in desc_embeddings.items()
    }
    return max(sims, key=sims.get)

Why this works: If the embedding space is geometrically meaningful — similar meanings → nearby vectors — then “A negative movie review expressing disappointment” should be close to actual negative reviews. The Sentence Transformer embedding space has exactly this property because it was trained to place semantically similar sentences near each other.

Label phrasing matters: The zero-shot approach is sensitive to how the class descriptions are written:

# Less effective:
"negative", "positive"

# More effective:
"A negative movie review expressing disappointment or criticism."
"A positive movie review expressing praise or enthusiasm."

# Even better (domain-specific):
"The reviewer disliked this film, finding it boring or poorly made."
"The reviewer enjoyed this film, praising its quality or entertainment value."

Typical accuracy with good descriptions: ~82–85% — nearly competitive with approaches that use thousands of labeled examples.

5.3.4.9. Approach 5: Generative Classification (Flan-T5)#

Flan-T5 is a sequence-to-sequence model trained to follow natural language instructions. For classification, we prompt it to generate the class label as text:

from transformers import pipeline

t5_pipeline = pipeline(
    "text2text-generation",
    model="google/flan-t5-small",
)

def classify_flan_t5(review):
    prompt = (
        "Is the following movie review positive or negative?\n\n"
        f"Review: {review}\n\n"
        "Answer with only 'positive' or 'negative'."
    )
    output = t5_pipeline(prompt, max_new_tokens=5)[0]["generated_text"]
    return 1 if "positive" in output.lower() else 0

Key insight: Flan-T5 treats all NLP tasks as text generation. Because it was instruction-tuned on tasks expressed as natural language questions and answers, it “knows” that “positive” and “negative” are the expected responses to this prompt format.

Risk: The model might generate unexpected text (e.g., “Positive, however the ending seems negative”). Robust parsing must handle these edge cases.

Typical accuracy: ~82–86% — comparable to embedding-based approaches, without requiring labeled data.

5.3.4.10. Approach 6: ChatGPT API Classification#

The highest-quality approach: route each review through a chat-based LLM with a structured prompt:

import openai

client = openai.OpenAI(api_key="YOUR_API_KEY")

def classify_chatgpt(review):
    messages = [
        {
            "role": "system",
            "content": "You are a helpful assistant that classifies movie reviews.",
        },
        {
            "role": "user",
            "content": (
                "Predict whether the following document is a positive or negative "
                "movie review:\n\n"
                f"{review}\n\n"
                "If it is positive return 1 and if it is negative return 0. "
                "Do not give any other answer."
            ),
        },
    ]
    response = client.chat.completions.create(
        messages=messages,
        model="gpt-3.5-turbo",
        temperature=0,
    )
    return int(response.choices[0].message.content.strip())

Typical accuracy: ~90–93% (GPT-3.5) / ~93–95% (GPT-4).

Trade-offs:

  • Highest accuracy — state-of-the-art quality without fine-tuning.

  • API cost: approximately $0.05 per 1,000 reviews with GPT-3.5.

  • Data privacy: reviews are sent to OpenAI’s servers.

  • Latency: 0.5–2 seconds per review vs. milliseconds for local approaches.

  • No offline use.

5.3.4.11.2. When to Use Each#

  • Use Approach 1 when you need a quick baseline with no data and no training, and can tolerate ~80% accuracy.

  • Use Approach 2 when you have a labeled training set and need fast inference at high accuracy.

  • Use Approach 3 when you have labeled data but can’t afford to train a model — or as a debugging baseline.

  • Use Approach 4 when you have no labeled data but can write good natural-language class descriptions.

  • Use Approach 5 when you need zero-shot performance and have GPU access but no API credits.

  • Use Approach 6 when you need maximum accuracy, have API budget, and data privacy is not a concern.

5.3.4.12. Error Analysis#

Aggregate accuracy hides important patterns. Systematic error analysis reveals where models fail:

5.3.4.12.1. Common Hard Examples#

  1. Irony and sarcasm

    “An absolute masterpiece… of wasted potential.” (Contains “masterpiece” but is negative — only contextual understanding catches this.)

  2. Qualified praise

    “While not perfect, the film offers genuine emotional moments.” (Positive despite prominent hedging language.)

  3. Critic-specific vocabulary

    “A lumpy, derivative genre exercise that squanders its premise.” (Negative, but unusual vocabulary may fall outside training distribution.)

  4. Domain mismatch

    “The ambiance was nice but the wait staff was inattentive.” (Yelp-style review, correctly negative, but movie-trained models may miss “wait staff”.)

5.3.4.12.2. Comparing Error Patterns Across Approaches#

# Find examples where approaches disagree
disagreements = [
    (i, test_texts[i], test_labels[i], y_pred_roberta[i], y_pred_logreg[i])
    for i in range(len(test_labels))
    if y_pred_roberta[i] != y_pred_logreg[i]
]

# Misclassified by both — these are the genuinely hard examples
both_wrong = [
    (test_texts[i], test_labels[i])
    for i in range(len(test_labels))
    if y_pred_roberta[i] != test_labels[i]
    and y_pred_logreg[i] != test_labels[i]
]

Examples that fool all approaches tend to be genuinely ambiguous, ironic, or domain-shifted — a useful signal for identifying data quality issues or annotation inconsistencies.

5.3.4.13. Exercises and discussion#

  1. Label description sensitivity For the zero-shot embedding approach (Approach 4), try four different natural-language descriptions for each class. How much does accuracy vary across description choices? What makes a description more or less effective?

  2. Embeddings + LogReg interpretability After training the Logistic Regression on sentence embeddings (Approach 2), the weight vector lives in 768-dimensional embedding space. Identify the 10 training reviews most aligned with the positive-class weight vector and the 10 most aligned with the negative-class weight vector. Do these represent the “most prototypically positive” and “most prototypically negative” reviews?

  3. Cost-accuracy trade-off You need to classify 100,000 reviews per day with 88% or better accuracy. Compare the monthly infrastructure cost of Approach 2 (GPU server for embedding + free LR inference) versus Approach 6 (ChatGPT API at $0.05/1K reviews). Under what accuracy threshold does Approach 6 become cost-effective despite its per-call fee?

5.3.4.14. Summary#

This section surveyed six fundamentally different approaches to text classification, unified by a common theme: the choice of representation determines what is possible without labeled data.

Key Insight

No single approach dominates across all dimensions. Embeddings + Logistic Regression offers the best accuracy-to-cost ratio when labeled data is available. Zero-shot embedding approaches are surprisingly competitive when they are not. ChatGPT API achieves the highest accuracy but introduces cost, latency, and privacy trade-offs. The right choice is always task-specific.
