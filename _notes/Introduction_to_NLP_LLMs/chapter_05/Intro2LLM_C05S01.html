5.3.1. Inside Transformer LLMs#

5.3.1.1. Learning Objectives#

By the end of this section, you should be able to:

  • Inspect the full PyTorch module hierarchy of a transformer LLM and describe what each named component does.

  • Identify the five major architectural components of a decoder-only transformer: token embeddings, decoder layers, self-attention, MLP blocks, and the language model head.

  • Trace a forward pass through Phi-3-Mini step by step, interpreting the shape of each intermediate tensor.

  • Explain what logits are, how they differ from probabilities, and how the softmax function converts one to the other.

  • Extract the top-\(k\) most probable next tokens from the model’s output distribution and interpret what each prediction means.

  • Explain what key-value (KV) caching is, why it exists, and quantify its impact on generation speed.

5.3.1.2. Why Look Inside?#

In Chapter 4 we treated LLMs as black boxes: prompts go in, text comes out. Understanding the internal architecture is essential for moving beyond that surface-level interaction:

  • Debugging unexpected outputs — knowing which component is responsible for which transformation.

  • Efficient deployment — understanding why KV caching makes generation 5–7× faster.

  • Fine-tuning — knowing which layers to freeze and which to update.

  • Prompt engineering — understanding why longer contexts affect generation quality and speed.

  • Research — interpreting model behavior through the lens of its architecture.

5.3.1.3. The Transformer Decoder Architecture#

All modern autoregressive LLMs (GPT, LLaMA, Phi, Mistral, Gemma) share the same fundamental architecture — a transformer decoder stack. The model processes a sequence of token IDs, applies 32 (or more) layers of self-attention and feed-forward transformations, and projects the final hidden state to a vocabulary distribution.

Key Definition — Autoregressive Generation

A model is autoregressive if it generates one token at a time, conditioning each new token on all previously generated tokens. Formally, given a sequence of tokens \(x_1, x_2, \ldots, x_t\), the model estimates the conditional probability \(P(x_{t+1} \mid x_1, \ldots, x_t)\) and samples (or selects) the next token from that distribution.

The primary model for this section is Phi-3-Mini-4K-Instruct — the same 3.8B-parameter model used in Chapter 4. Loading it requires only a few lines:

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

# Reuse the tokenizer if it already exists in the notebook to avoid re-downloading.
if "tokenizer" not in globals():
    tokenizer = AutoTokenizer.from_pretrained(
        "microsoft/Phi-3-mini-4k-instruct")

# Load Phi-3 Mini for causal language modeling.
# - device_map="cuda": place weights on GPU
# - torch_dtype="auto": choose a memory-efficient dtype automatically
# - trust_remote_code=False: use only library-native model code
model = AutoModelForCausalLM.from_pretrained(
    "microsoft/Phi-3-mini-4k-instruct",
    device_map="cuda",
    torch_dtype="auto",
    trust_remote_code=False,
)

Here device_map="cuda" places model weights on the GPU, and torch_dtype="auto" lets the library choose the most memory-efficient floating-point precision (typically bfloat16 on modern hardware).

Key Definition — Hidden Dimension (\(d_\text{model}\))

The hidden dimension \(d_\text{model} = 3072\) is the width of every intermediate representation in Phi-3-Mini. Every token is represented as a vector of this size throughout the entire forward pass, from the embedding lookup all the way to the final layer normalization.

Key Definition — RMSNorm

Root Mean Square Normalization (RMSNorm) is a simplified alternative to Layer Normalization. Given a hidden vector \(\mathbf{h} \in \mathbb{R}^d\), it rescales each element by the inverse of the root-mean-square of the vector:

(5.1)#\[\begin{equation} \text{RMSNorm}(\mathbf{h}) = \frac{\mathbf{h}}{\sqrt{\frac{1}{d}\sum_{i=1}^{d} h_i^2 + \epsilon}} \odot \boldsymbol{\gamma} \end{equation}\]

where \(\boldsymbol{\gamma}\) is a learned scale parameter and \(\epsilon\) is a small constant (here \(10^{-5}\)) for numerical stability. RMSNorm is computationally cheaper than Layer Norm because it omits the mean-centering step.

5.3.1.4.1. Parameter Distribution#

The following snippet counts parameters per component and highlights where most of the model’s capacity resides:

def count_params(module):
    """Count total number of parameters in a module."""
    total = sum(p.numel() for p in module.parameters())
    return total

# Display parameter counts for each major component
components = {
    "Embedding layer (embed_tokens)": model.model.embed_tokens,
    "All 32 decoder layers":          model.model.layers,
    "Final norm":                     model.model.norm,
    "LM head":                        model.lm_head,
    "Full model":                     model,
}

for name, module in components.items():
    print(f"{name:<35} {count_params(module):>15,}")

Embedding layer (embed_tokens)           98,500,608
All 32 decoder layers                 3,624,075,264
Final norm                                    3,072
LM head                                  98,500,608
Full model                            3,821,079,552

Key observations:

  • The embedding layer and LM head each contain approximately 98.5 M parameters (\(32{,}064 \times 3{,}072\)).

  • In Phi-3, these weights are tied — the same matrix serves as both input embedding lookup and output logit projection. This saves 98.5 M parameters with no measurable quality loss.

  • The 32 decoder layers together contain approximately 3.3 B of the 3.8 B total parameters, meaning the vast majority of the model’s learned capacity lives in those stacked transformer blocks.

5.3.1.5. Forward Pass Step by Step#

Instead of using pipeline, we can run the forward pass manually to observe the exact tensor shapes at each stage. This makes the data flow concrete and inspectable.

prompt = "The capital of Japan is"

# Step 1: Tokenize
input_ids = tokenizer(prompt, return_tensors="pt").input_ids.to("cuda")
# Shape: (1, 5)  — batch=1, seq_len=5

# Step 2: Run through the transformer backbone (32 layers)
model_output = model.model(input_ids)
# model.model = Phi3Model (backbone without LM head)
# Output hidden state shape: (1, 5, 3072)

# Step 3: Project through the LM head
lm_head_output = model.lm_head(model_output[0])
# Shape: (1, 5, 32064)  — a score for every vocabulary token at every position

# Step 4: Extract next-token prediction from the last position
last_logits = lm_head_output[0, -1]   # Shape: (32064,)
token_id_greedy = last_logits.argmax(-1)
print(f"Predicted next token: '{tokenizer.decode(token_id_greedy)}'")

Why the last position? In causal (left-to-right) generation, each token position can only attend to itself and the tokens that came before it — this is called causal masking. Position 0 has attended only to itself; position 4 has attended to all five tokens. The last position’s logit vector therefore contains the most context-informed prediction of what comes next, so we always read off the next token from there.

Key Definition — Causal Masking

Causal masking (also called an autoregressive mask) prevents each token from attending to future positions during self-attention. Concretely, the attention weight from position \(i\) to position \(j\) is set to \(-\infty\) (before softmax) whenever \(j > i\). This enforces the left-to-right generation order.

5.3.1.6. Logits, Probabilities, and the Vocabulary Distribution#

Logits are the raw, unnormalized scores output by the LM head. They can be any real number — positive, negative, large, or small. The softmax function converts them into a proper probability distribution over all \(V = 32{,}064\) vocabulary tokens.

Key Definition — Softmax

Given a logit vector \(\mathbf{z} \in \mathbb{R}^V\), the softmax function produces a probability distribution \(\mathbf{p} \in \mathbb{R}^V\) where every entry is non-negative and the entries sum to 1:

(5.2)#\[\begin{equation} p_i = \frac{e^{z_i}}{\sum_{j=1}^{V} e^{z_j}}, \quad \sum_{i=1}^{V} p_i = 1 \end{equation}\]

Tokens with higher logits receive higher probabilities, but even tokens with negative logits receive a small non-zero probability.

A concrete example for the prompt "The capital of Japan is":

Logit for "Tokyo"  :  47.25  (highest)
Logit for "Kyoto"  :  43.10
Logit for "a"      :  39.80
...
Logit for "banana" : -12.50  (very unlikely)

After softmax:
P("Tokyo")  ≈ 0.836  (83.6% probability)
P("Kyoto")  ≈ 0.009  (0.9%)
...
Sum of all 32,064 probabilities = 1.000

import torch.nn.functional as F

probs = F.softmax(last_logits, dim=-1)
print(f"Sum of all probs: {probs.sum().item():.6f}")  # → 1.000000

k = 10
top_probs, top_ids = torch.topk(probs, k)
for rank, (token_id, prob) in enumerate(zip(top_ids, top_probs), 1):
    token_str = tokenizer.decode([token_id])
    print(f"  {rank}. '{token_str}' — {prob.item():.4f}")

5.3.1.6.1. The Role of Temperature#

During inference, temperature reshapes the logit distribution before applying softmax. All logits are divided by a scalar \(T > 0\):

(5.3)#\[\begin{equation} \tilde{z}_i = \frac{z_i}{T} \end{equation}\]

Temperature

Effect

\(T = 0.1\)

Very peaked: top token gets near-100% probability → near-deterministic output

\(T = 1.0\)

Original unmodified distribution (default)

\(T = 2.0\)

Flatter: more probability mass on lower-ranked tokens → more diverse but less coherent

\(T \to 0\)

Equivalent to argmax (greedy decoding)

\(T \to \infty\)

Uniform random selection over all 32,064 vocabulary tokens

Intuitively, low temperature makes the model more confident (and repetitive), while high temperature makes it more creative (and error-prone). In practice, values between \(T = 0.7\) and \(T = 1.2\) are most common.

5.3.1.7. Hidden States Across Layers#

The output_hidden_states=True flag causes the model to return the hidden state tensor after every decoder layer. This lets us observe how representations evolve through the network — an important diagnostic for understanding what the model has learned and where.

outputs = model.model(input_ids, output_hidden_states=True)
# outputs.hidden_states is a tuple of (num_layers + 1) tensors
# Shape of each: (batch, seq_len, 3072)
# Index 0 = raw token embeddings; indices 1–32 = after each decoder layer

for layer_idx, hidden in enumerate(outputs.hidden_states):
    last_token = hidden[0, -1]   # last token's representation at this layer
    l2_norm = last_token.norm().item()
    print(f"Layer {layer_idx:>2}: last-token L2 norm = {l2_norm:.2f}")

The L2 norm of the last-token representation tends to grow through the layers as each decoder block adds a residual update:

(5.4)#\[\begin{equation} \mathbf{h}^{(\ell)} = \mathbf{h}^{(\ell-1)} + \Delta_\text{attn}^{(\ell)} + \Delta_\text{MLP}^{(\ell)} \end{equation}\]

where \(\mathbf{h}^{(\ell-1)}\) is the input to layer \(\ell\), \(\Delta_\text{attn}\) is the self-attention update, and \(\Delta_\text{MLP}\) is the feed-forward update. This residual (skip) connection ensures that gradient can flow directly back to early layers during training and that early representations are preserved as later layers refine them.

The final layer’s representation contains the richest context-aware encoding, which the LM head then projects into vocabulary space.

5.3.1.8. The Language Model Head#

The LM head is a single linear layer — the simplest possible transformation:

hidden_state: (batch, seq_len, 3072)   ← rich contextual representation
      ↓
  Linear(3072 → 32064, bias=False)     ← learned projection matrix W
      ↓
logits: (batch, seq_len, 32064)        ← unnormalized score for each vocabulary token

Formally, for a hidden vector \(\mathbf{h} \in \mathbb{R}^{d_\text{model}}\), the LM head computes:

(5.5)#\[\begin{equation} \mathbf{z} = W\,\mathbf{h}, \quad W \in \mathbb{R}^{V \times d_\text{model}} \end{equation}\]

No activation function, no bias, no normalization — just a matrix multiplication. The power comes entirely from the rich 3,072-dimensional hidden state built up through 32 transformer layers.

Weight tying connects the embedding lookup table to the LM head using the same matrix \(W\). When embedding “Tokyo,” the model retrieves a 3,072-dim row vector representing “Tokyo’s meaning.” When predicting “Tokyo” as the next token, the LM head scores how similar the hidden state is to that same vector. High similarity → high logit for “Tokyo.” This symmetry saves 98.5 M parameters with no measurable quality loss.

5.3.1.9. Key-Value Caching#

When generating text autoregressively, the model must produce one token at a time. Without caching, every generation step recomputes the attention keys and values for all previous tokens from scratch.

Key Definition — Self-Attention Keys and Values

In self-attention, each token’s hidden vector \(\mathbf{h}\) is projected into three separate vectors:

  • Query \(\mathbf{q} = W_Q \mathbf{h}\) — what this token is looking for.

  • Key \(\mathbf{k} = W_K \mathbf{h}\) — what this token advertises about itself.

  • Value \(\mathbf{v} = W_V \mathbf{h}\) — the information this token contributes if attended to.

The attention score between position \(i\) (query) and position \(j\) (key) is:

(5.6)#\[\begin{equation} a_{ij} = \frac{\mathbf{q}_i^\top \mathbf{k}_j}{\sqrt{d_k}} \end{equation}\]

where \(d_k\) is the key dimension. After softmax over \(j\), these scores weight the values \(\mathbf{v}_j\) to produce the output for position \(i\).

Without caching, every step recomputes \(\mathbf{k}\) and \(\mathbf{v}\) for all prior tokens:

Without KV cache (generating N tokens from a prompt of length P):

Step 1: compute K, V for all P tokens → predict token 1
Step 2: recompute K, V for P+1 tokens → predict token 2  (P redundant)
Step 3: recompute K, V for P+2 tokens → predict token 3  (P+1 redundant)
...
Total wasted computation: O(n²) for n tokens generated

With KV caching, computed keys and values are stored after each step. On subsequent steps, only the new token’s \(\mathbf{k}\) and \(\mathbf{v}\) need to be computed and appended:

With KV cache:

Step 1: compute K, V for all P tokens → SAVE to cache → predict token 1
Step 2: compute K, V for NEW token only → CONCAT with cache → predict token 2
Step 3: compute K, V for NEW token only → CONCAT with cache → predict token 3
...
Total complexity: O(n) per step instead of O(n²)

The practical speedup is substantial. For Phi-3-Mini generating 100 tokens:

import time

# Without cache
start = time.time()
output_no_cache = model.generate(input_ids, max_new_tokens=100, use_cache=False)
time_no_cache = time.time() - start

# With cache (default)
start = time.time()
output_cache = model.generate(input_ids, max_new_tokens=100, use_cache=True)
time_cache = time.time() - start

print(f"Without cache: {time_no_cache:.2f}s")
print(f"With cache   : {time_cache:.2f}s")
print(f"Speedup      : {time_no_cache / time_cache:.1f}×")

Typical speedup: 5–7× for 100 tokens, growing with sequence length because the cache eliminates an increasingly large fraction of redundant computation. The trade-off is memory: the KV cache grows linearly with sequence length and the number of layers, which is one reason long-context inference requires significantly more GPU memory.

The table below summarizes every stage of the forward pass, from raw text to a predicted next token:
