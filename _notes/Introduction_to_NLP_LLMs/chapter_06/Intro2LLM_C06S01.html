6.3.1. Foundations of Prompt Engineering#

6.3.1.1. Learning objectives#

By the end of this section, you should be able to:

  • Define prompt engineering and explain why instruction-tuned LLMs are sensitive to prompt phrasing.

  • Identify the seven core ingredients of a well-engineered prompt and explain the role of each.

  • Construct basic prompts using the chat template format with role and content fields.

  • Explain how temperature, top_p, and do_sample control generation behavior and predict their effect on output diversity.

  • Inspect the raw chat template applied to a message list and identify the model-specific special tokens.

6.3.1.2. What Is Prompt Engineering?#

Prompt engineering [Liu et al., 2023] is the practice of systematically designing, structuring, and refining the input given to a language model to reliably elicit the desired output. Unlike traditional software development where inputs are strictly constrained by data types (e.g., integers, booleans), LLMs accept open-ended natural language. This means that the exact same intent, phrased differently, can yield drastically different results in quality, formatting, and factual accuracy.

A well-engineered prompt serves as a practitioner’s primary lever for:

  • Steering model behavior without modifying underlying model weights (avoiding expensive fine-tuning).

  • Reducing hallucinations and keeping the model anchored to specific, provided context.

  • Enforcing deterministic output formats (such as strict JSON schemas, YAML, or structured Markdown) for downstream application parsing.

  • Deconstructing complex reasoning tasks into sequential, traceable steps.

  • Adapting general-purpose models to highly specialized domain tasks (e.g., legal analysis, medical coding).

6.3.1.2.1. Why Are LLMs Sensitive to Phrasing?#

The sensitivity of LLMs to prompt phrasing arises directly from how they are trained. During Supervised Fine-Tuning (SFT) and subsequent alignment phases like Reinforcement Learning from Human Feedback (RLHF) [Ouyang et al., 2022], the model learns statistical associations between specific prompt design patterns and high-quality responses.

When an input matches these learned structural patterns (such as standard chat templates or explicit instruction headers), it activates the model’s optimal generation pathways. Conversely, minor deviations, ambiguous phrasing, or missing structural tokens can cause the model to fall back on generic text-completion patterns, leading to unpredictable, degraded, or off-topic outputs.

6.3.1.3. The Prompt Engineering Landscape#

The techniques covered in this chapter span a spectrum from foundational execution to advanced cognitive frameworks:

Foundational
  ├── Chat template formatting & role assignment (System, User, Assistant)
  ├── Generation parameter control (temperature, top_p, do_sample)
  └── The seven core prompt ingredients

Intermediate
  ├── Advanced multi-ingredient prompt construction
  ├── Precision engineering: Specificity & hallucination mitigation
  └── Chain prompting (breaking monolithic tasks into sequential API calls)

Advanced
  ├── In-context learning (zero-shot, one-shot, and few-shot exemplars)
  ├── Explicit reasoning: Chain-of-Thought (CoT) & self-consistency 
  ├── Advanced search: Tree-of-Thought (ToT) for multi-path problem solving
  └── Constrained decoding: Grammar-constrained structured generation

6.3.1.4. Model Loading and Pipeline Setup#

All experiments in this chapter use Phi-3-Mini-4K-Instruct [Abdin and others, 2024] as the primary model, loaded through the Hugging Face transformers library [Wolf et al., 2020] and PyTorch [Paszke et al., 2019]. The setup mirrors the environment configuration from Chapter 4:

Show code cell source

Hide code cell source

 
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, pipeline

model = AutoModelForCausalLM.from_pretrained(
    "microsoft/Phi-3-mini-4k-instruct",
    device_map="cuda",
    dtype="auto",
    trust_remote_code=False,
)

tokenizer = AutoTokenizer.from_pretrained("microsoft/Phi-3-mini-4k-instruct")

pipe = pipeline(
    "text-generation",
    model=model,
    tokenizer=tokenizer,
    max_new_tokens=200,
    return_full_text=False,
    do_sample=True,
    temperature=0.7,
)

Device set to use cuda

Engineering Note: Setting trust_remote_code=False ensures that we rely strictly on native Hugging Face library implementations rather than executing arbitrary custom code files hosted on the model repository. This is a critical security best practice in production machine learning workflows.

6.3.1.5. The Chat Template Format#

Instruction-tuned models are rarely given raw, unstructured text during conversational tasks. Instead, they expect inputs structured as a list of message dictionaries, where each dictionary represents a single conversational turn defined by a strict schema containing a role and its corresponding content.

Show code cell source

Hide code cell source

 
messages = [
    {"role": "user", "content": "Create a funny joke about tigers."}
]

output = pipe(messages)
print(output[0]["generated_text"])

 Why don't tigers ever get lost? Because they always follow their instincts!

To effectively steer a model, practitioners utilize three distinct ecosystem roles:

Table 6.1 Ecosystem Roles and Usage Strategies#

Role

Purpose & Usage Strategy

system

Sets the foundational persona, behavioral constraints, tone, and safety guardrails. It acts as the “metacognitive layer” that persists across the session.

user

Represents the human operator turn—providing the explicit task, context, query, or runtime instructions.

assistant

Represents the model’s generated response. It can also be pre-filled by the developer to forcefully bias or structure the model’s starting output.

6.3.1.6. Inspecting the Raw Chat Template#

Behind the scenes, language models do not natively understand dictionary formats or abstract roles. Before inference occurs, the pipeline invokes a model-specific chat template compiled into the tokenizer’s configuration. This template parses the message array and injects exact control tokens that act as rigid structural boundaries for the neural network.

Printing the formatted string reveals the precise special tokens mandated by Phi-3’s architecture:

Show code cell source

Hide code cell source

 
prompt_formatted = pipe.tokenizer.apply_chat_template(messages, tokenize=False)
print(prompt_formatted)

<|user|>
Create a funny joke about tigers.<|end|>
<|endoftext|>

6.3.1.6.1. Why Inspecting Special Tokens Matters#

Every open-source model family (e.g., Llama, Mistral, Phi) utilizes entirely different special control tokens (such as <|user|> vs [INST]).

  • If these control tokens are omitted, corrupted, or manually hardcoded incorrectly as regular text strings, the model will fail to recognize where the user’s instructions end and where its own response should begin.

  • Programmatically generating these prompts via apply_chat_template ensures perfect alignment with the model’s pre-training and fine-tuning boundaries.

The model never processes raw Python dictionaries directly. Instead, it reads a singular compiled text string interspersed with strict structural markers. For the Phi-3-Mini architecture, these markers are defined as follows:

6.3.1.7. Generation Parameters#

The parameters passed to a generation pipeline or API call alter the mathematical probability distribution over the model’s vocabulary at each successive step.

6.3.1.7.1. do_sample and temperature#

When do_sample=False, the model uses greedy decoding, meaning it strictly selects the single token with the highest log-probability mass at every step (\(t\)). While highly deterministic, pure greedy decoding can lead to repetitive loops or overly sterile text in creative tasks.

When do_sample=True, the model samples stochastically from its vocabulary based on their calculated probability distribution. The temperature parameter (\(T\)) scales these raw log-probabilities (logits) before applying the Softmax function:

(6.1)#\[\begin{equation} \text{Softmax}(z_i) = \frac{e^{z_i / T}}{\sum_{j} e^{z_j / T}} \end{equation}\]

Show code cell source

Hide code cell source

 
# High temperature — flattens the distribution, increasing diversity and surprise
output = pipe(messages, do_sample=True, temperature=1.0)

# Lower temperature — peaks the distribution around the most likely tokens, increasing focus
output = pipe(messages, do_sample=True, temperature=0.3)
Table 6.3 Generation Parameter Configurations#

Parameter Configuration

Mathematical Effect

Best Used For

do_sample=False

Greedy Decoding: Zero randomness; \(T\) is ignored.

Factual Q&A, JSON/YAML data extraction, deterministic code execution.

temperature=0.1 to 0.3

Focused Sampling: Sharpens the distribution; heavily favors the top 1-2 choices.

Code generation, complex mathematical reasoning, rigid text formatting.

temperature=0.7 to 0.9

Balanced Sampling: Maintains original token distribution dynamics.

General writing, summarization, conversational agents, creative prose.

temperature=1.2 to 1.5

High Randomness: Flattens distribution; lower-probability “creative” words surface.

Brainstorming, ideation, expansive fiction writing.

6.3.1.8. top_p (Nucleus Sampling)#

Greedy decoding and low-temperature sampling can still assign minor probability mass to an unreliable “long tail” of low-quality or irrelevant tokens. If selected, these tail tokens can permanently derail the coherence of the remaining sequence.

Nucleus sampling [Holtzman et al., 2020] solves this by dynamically truncating the tail before sampling occurs. The parameter top_p restricts the model’s choices to the smallest subset of top tokens whose cumulative probability sum meets or exceeds the threshold \(p\):

Show code cell source

Hide code cell source

 
# Sample only from the top 50% of the probability mass — highly focused
output = pipe(messages, do_sample=True, top_p=0.5, temperature=0.7)

# Sample from the top 90% of the probability mass — allows broader vocabulary expression
output = pipe(messages, do_sample=True, top_p=0.9, temperature=0.7)

6.3.1.9. The Seven Core Prompt Ingredients#

A production-grade prompt rarely consists of a single instruction. To guarantee reliability and structure, a well-engineered prompt typically combines some or all of seven core ingredients. Each ingredient serves as a specific engineering lever designed to mitigate unique model failure modes:

Table 6.4 The Seven Core Prompt Ingredients#

#

Ingredient

Technical Purpose

Mitigated Failure Mode

Example

1

Role

Calibrates the semantic space and token-association biases of the network.

Generic, surface-level, or uninspired responses.

"You are an expert data scientist."

2

Context

Pins down the background boundaries, target audience assumptions, and operational constraints.

Scope creep or mismatched complexity levels.

"The user is a first-year undergraduate student."

3

Task

Declarative statement of the core transformation or execution goal (the mandatory action).

Aimless text generation or conversational drifting.

"Summarize the following article."

4

Format

Dictates structural parsing paradigms (JSON keys, markdown headers, array lists).

Downstream code parsing failures.

"Respond in exactly three bullet points."

5

Constraints

Hard boundaries defining what the model must not include or do.

Hallucinations, information leaks, or technical jargon.

"Do not use any mathematical notation."

6

Examples

In-context learning sequences (few-shot) demonstrating input-to-output patterns.

Pattern drift or misalignment with complex stylistic norms.

[Input/Output Exemplars]

7

Tone

Calibrates vocabulary selection for specific social/professional registers.

Inappropriate or abrasive conversational styling.

"Use formal academic language."

6.3.1.9.1. Assembling a Minimal Viable Prompt (MVP)#

You do not need to use all seven ingredients in every prompt. Overloading a prompt with unnecessary ingredients can cause attention dilution, where the model misses critical instructions because its attention weights are spread across too many tokens.

A highly effective baseline prompt typically balances Role, Task, Format, and Constraints:

Show code cell source

Hide code cell source

 
messages = [
    {
        "role": "system",
        "content": (
            "You are a helpful NLP instructor. "         # 1. Role
            "Explain concepts clearly and concisely."     # 7. Tone
        ),
    },
    {
        "role": "user",
        "content": (
            "Explain what a transformer is "             # 3. Task
            "in exactly three bullet points, "           # 4. Format
            "without using mathematical notation."       # 5. Constraint
        ),
    }
]

output = pipe(messages)
print(output[0]["generated_text"])

 - A transformer is an electrical device that transfers electrical energy between two or more circuits through electromagnetic induction.

- It typically consists of two coils of wire, known as the primary and secondary windings, which are wrapped around a common core, often made of iron or ferrite.

- The transformer operates on the principle of Faraday's law of induction, which states that a change in magnetic flux through a coil will induce a voltage across the coil, allowing for the efficient transfer of energy.

6.3.1.9.2. Anatomy of Execution#

When the above payload is executed, the apply_chat_template weaves these blocks together behind the scenes. The model assigns heavy attention weights to the system tokens to bound its persona before processing the execution directives inside the user token blocks, yielding a clean, constrained response that satisfies all criteria simultaneously.

Not all seven ingredients are needed for every task. The key is to include the ingredients that prevent the specific failure modes you observe: vague outputs → add format; off-topic responses → add constraints; inconsistent style → add role and tone.

6.3.1.10. Precision Engineering & Specificity#

A common failure mode in naive prompting is semantic ambiguity—using broad, subjective adjectives (e.g., “write a short essay”, “make it professional”) that leave the model’s token-selection space wide open. Precision engineering means replacing vague instructions with clear, quantifiable metrics and crisp semantic anchors.

6.3.1.10.1. Quantifiable Metrics over Vague Directives#

Instead of relying on terms like “brief” or “detailed,” which scale unpredictably based on a model’s pre-training biases, use exact constraints:

  • Vague: "Summarize this report briefly."

  • Precise: "Summarize this report in fewer than 150 words." or "Summarize this report in exactly three sentences."

# Vague phrasing leads to highly variable output length
vague_messages = [{"role": "user", "content": "Write a short paragraph about photosynthesis."}]

# Precise phrasing restricts the generation space tightly
precise_messages = [{"role": "user", "content": "Write an explanation of photosynthesis restricted to exactly two sentences, totaling under 50 words."}]

6.3.1.10.2. Eliminating Intent Ambiguity#

When a task has multiple logical interpretations, the model will pick the statistically most likely path, which may not align with your goal. Explicitly define your assumptions and target audience to lock down the context:

  • Ambiguous: "Explain how inflation works."

  • Context-Anchored: "Explain how inflation works from the perspective of Keynesian economics, targeting an audience of high school students with no prior background in finance."

6.3.1.11. Handling Negative Constraints#

Large Language Models generate text sequentially by predicting the next token based on all preceding tokens. Because of this forward-generation mechanism, explicitly telling a model not to do something can sometimes backfire:

The Pink Elephant Problem: Mentioning a phrase inside a negative constraint (e.g., "Do not mention the word 'elephant'") heavily primes those exact token IDs in the model’s active attention window, accidentally increasing their probability mass.

6.3.1.11.1. Transforming Negative Constraints into Positive Directives#

While modern instruction-tuned models like Phi-3 handle negative constraints much better than older base models, the most robust way to ensure compliance is to pair a negative constraint with an explicit positive alternative action.

Table 6.5 Transforming Negative Constraints into Positive Directives#

Weak Negative Constraint

Robust Positive Directive Alternative

"Do not use complex technical jargon."

"Write exclusively for an 8th-grade reading level. Replace any technical term with a simple, everyday analogy."

"Don't write an intro or conclusion."

"Begin your response directly with the raw data table. Terminate generation immediately after the final data row."

"Do not include any personal bias."

"Maintain an objective, neutral tone. Present arguments for both sides with equal word count and weight."

6.3.1.12. Delimiters#

When a prompt contains both operational instructions (“Summarize this text”) and runtime data variables (“The text to summarize”), models can suffer from instruction-data conflation. In severe cases, malicious runtime data can hijack the prompt entirely (known as Prompt Injection).

Delimiters act as visual and structural boundary walls, isolating data payloads from core logic instructions.

Show code cell source

Hide code cell source

 
# A robust prompt architecture utilizing text delimiters
messages = [
    {
        "role": "user",
        "content": (
            "You are a text classification assistant. Analyze the customer review enclosed "
            "within triple backticks (```). Classify its sentiment as either POSITIVE or NEGATIVE.\n\n"
            "Review:\n"
            "```\n"
            "The product arrived three days late, and the packaging was completely torn. "
            "However, once I plugged it in, it worked flawlessly and exceeded my expectations!\n"
            "```"
        )
    }
]

6.3.1.12.1. Standard Delimiter Conventions#

Choose a distinct structural delimiter pattern and explicitly reference it within your instructions:

  • Triple Backticks: [data] (highly recommended for code and markdown blocks)

  • XML-Style Tags: <review> [data] </review> (excellent for complex, multi-variable payloads because open/close tags explicitly track structural depth)

  • Double Quotes or Capital Headers: --- DATA START --- (useful for cleanly separating sprawling text documents)

By standardizing your boundaries, you ensure that the model processes variable runtime data purely as an input string rather than parsing it as executable instructions.

6.3.1.13. Exercises and discussion#

  1. Template exploration Load two different instruction-tuned models (e.g., Phi-3-Mini and a Llama-3 model if available) and call apply_chat_template on the same message list. Compare the special tokens used by each model. Why would different models use different chat templates?

  2. Temperature sensitivity Generate five responses to the same creative prompt (e.g., “Write an opening sentence for a science fiction story set on Mars”) at temperatures 0.1, 0.5, 1.0, 1.5, and 2.0. At which temperature do responses become incoherent? At which are they too repetitive?

  3. Seven-ingredient audit Take a prompt you have previously written for an LLM (in any context). Identify which of the seven ingredients it contains and which are missing. Rewrite the prompt to include at least five ingredients and compare the output quality.
