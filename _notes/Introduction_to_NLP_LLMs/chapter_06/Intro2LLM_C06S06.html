6.3.6. Prompt Engineering in Practice#

6.3.6.1. Learning objectives#

By the end of this section, you should be able to:

  • Use a principled decision framework to select the appropriate prompting technique for a given task.

  • Apply prompt engineering techniques to a diverse set of NLP tasks: classification, extraction, summarization, and generation.

  • Iterate systematically on a prompt — identify the failure mode, select the appropriate fix, and re-evaluate.

  • Describe the prompting technique selection guide and explain the criteria that drive each decision.

6.3.6.2. The Technique Selection Framework#

After covering six prompting strategies — basic prompting, advanced prompt construction, zero-shot ICL, few-shot ICL, Chain-of-Thought, and structured generation — the practical challenge is: which technique should you use for a given task?

The following decision framework structures that choice:

STEP 1: Is the output structured (JSON, YAML, specific format)?
  Yes → Use one-shot + validation loop; if reliability critical, use grammar-constrained
  No  → Continue to step 2

STEP 2: Does the task require multi-step reasoning?
  Yes → Use Chain-of-Thought (zero-shot CoT for most tasks, few-shot CoT for math/logic)
  No  → Continue to step 3

STEP 3: Is the output pattern complex or unusual?
  Yes → Use few-shot (2–5 examples demonstrating the pattern)
  No  → Continue to step 4

STEP 4: Is output quality with a basic prompt acceptable?
  Yes → Use zero-shot (task + format specification)
  No  → Add role, context, constraints; re-evaluate (advanced prompt construction)

STEP 5: Is the task too complex for a single prompt?
  Yes → Use chain prompting (decompose into sequential steps)
  No  → Optimize the single prompt further

# Chain: topic → outline → section drafts → final assembly
def generate_blog_post(topic: str) -> str:
    # Step 1: Generate outline
    outline_messages = [
        {
            "role": "system",
            "content": "You are a technical blog writer.",
        },
        {
            "role": "user",
            "content": (
                f"Create a 4-section outline for a blog post about: {topic}\n"
                "Format: Section 1: [title]\nSection 2: [title]\n..."
            ),
        },
    ]
    outline = pipe(outline_messages)[0]["generated_text"]

    # Step 2: Expand each section
    draft_messages = [
        {
            "role": "system",
            "content": "You are a technical blog writer. Write clearly for a data science audience.",
        },
        {
            "role": "user",
            "content": (
                f"Using this outline:\n{outline}\n\n"
                f"Write the full blog post about '{topic}'. "
                "Each section should be 2–3 paragraphs. "
                "Use concrete examples throughout."
            ),
        },
    ]
    return pipe(draft_messages, max_new_tokens=800)[0]["generated_text"]

6.3.6.4. Prompt Iteration Workflow#

When a prompt does not produce the desired output, follow this diagnostic workflow:

1. OBSERVE the failure
   What exactly went wrong?
   - Wrong format → add/improve format specification
   - Off-topic → add context and constraints
   - Hallucination → add grounding instructions, source material
   - Inconsistent → add examples (zero→one→few-shot)
   - Reasoning error → add Chain-of-Thought trigger
   - Structured output broken → add one-shot example or grammar constraints

2. HYPOTHESIZE the cause
   Which ingredient is missing or underspecified?

3. FIX with the minimal change
   Add only what is needed — over-engineered prompts can degrade performance

4. EVALUATE on multiple inputs
   Test on at least 10 diverse inputs before declaring success

5. DOCUMENT what worked
   Treat effective prompts as engineering artifacts — version control them

6.3.6.5. Prompting Technique Summary#

Technique

When to use

Key implementation

Zero-shot [Brown et al., 2020]

Common tasks, simple format

Task + format constraint in user message

One-shot [Brown et al., 2020]

Structured output, unusual schema

One example as user/assistant pair

Few-shot [Brown et al., 2020]

Classification, complex patterns

3–5 diverse examples

Chain-of-Thought [Wei et al., 2022]

Multi-step reasoning, math, logic

“Let’s think step by step” [Kojima et al., 2022] or worked example

Self-consistency [Wang et al., 2023]

High reliability on reasoning tasks

5 samples + majority vote

Tree-of-Thought [Yao et al., 2023]

Multi-path planning, exploration

Multiple expert perspectives in prompt

Chain prompting [Wu et al., 2022]

Complex multi-stage tasks

Sequential LLM calls, output feeds next

Grammar-constrained [Geng et al., 2023]

Guaranteed JSON/structured output

BNF grammar + llama-cpp-python [Gerganov and contributors, 2023]

6.3.6.6. Exercises and discussion#

  1. Framework application Choose five tasks from different domains: (a) tweet sentiment classification, (b) medical entity extraction, (c) multi-hop question answering, (d) code generation, and (e) product description writing. For each, apply the decision framework step-by-step and justify the technique you selected before implementing and testing it.

  2. Iterative improvement case study Take a prompt that produces mediocre output and iterate through five improvements, documenting each change and its effect on output quality. Use the diagnostic workflow above to guide each iteration. What is the cumulative improvement from the initial prompt to the final version?

  3. Prompting vs. fine-tuning For what types of tasks does prompt engineering reach its ceiling — where no amount of prompting improvement will produce acceptable output quality? List three such tasks and explain what property makes them resistant to prompting. For each, what fine-tuning approach would you consider instead?
