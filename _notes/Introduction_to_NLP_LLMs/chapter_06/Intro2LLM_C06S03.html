6.3.3. In-Context Learning: Zero-Shot, One-Shot, and Few-Shot#

6.3.3.1. Learning objectives#

By the end of this section, you should be able to:

  • Define in-context learning (ICL) and explain why it does not modify model weights.

  • Apply zero-shot prompting and identify when it is sufficient.

  • Construct one-shot and few-shot prompts by including representative input-output examples.

  • Explain how the quality, diversity, and order of examples affect few-shot performance.

  • Select the appropriate ICL strategy (zero-shot, one-shot, few-shot) for a given task and resource constraint.

6.3.3.2. What Is In-Context Learning?#

In-context learning (ICL) [Brown et al., 2020] teaches the model what output you want by showing examples directly in the prompt — no fine-tuning required. The number of examples defines the zero/one/few-shot taxonomy, first formalized alongside GPT-3 [Brown et al., 2020]:

Zero-shot:  No examples; the model infers output format from the instruction alone
One-shot:   One example; provides a schema for the model to follow
Few-shot:   Two or more examples; shapes behavior for consistent, complex output patterns

ICL is fundamentally different from fine-tuning: the model’s weights are never updated. The examples in the prompt are processed as ordinary context — but they activate strong associative patterns from the model’s training, causing it to mirror the demonstrated format and style.

import torch

6.3.3.3. Zero-Shot Prompting#

Zero-shot prompting asks the model to perform a task with no demonstration of the desired output. Success depends entirely on whether the task and format are familiar from the model’s training data.

Show code cell source

Hide code cell source

 
messages = [
    {
        "role": "system",
        "content": "You are a data extraction assistant. Respond only with JSON.",
    },
    {
        "role": "user",
        "content": (
            "Extract the following fields from the text and return them as a JSON object: "
            "{name, organization, role, location}. Return ONLY the JSON.\n\n"
            "Text: Dr. Sarah Chen, a professor at MIT's CSAIL lab in Cambridge, "
            "leads research on neural machine translation."
        ),
    },
]
output = pipe(messages, do_sample=False, temperature=None)[0]["generated_text"]
print(output)

 ```json

{

  "name": "Dr. Sarah Chen",

  "organization": "MIT's CSAIL lab",

  "role": "professor",

  "location": "Cambridge"

}

```

Zero-shot works well when:

  • The task type is common and well-represented in instruction-tuning data (Q&A, summarization, classification).

  • The output format is simple (yes/no, a single number, a short sentence).

  • The model is sufficiently capable (7B+ instruction-tuned).

Zero-shot often fails when:

  • The output schema is unusual or highly specific.

  • The model must infer implicit requirements from the instruction alone.

  • The task requires domain-specific reasoning not covered in training.

6.3.3.4. One-Shot Prompting#

One-shot prompting provides a single input-output example before the query. The example acts as a schema: the model learns the exact output format and content structure from one demonstration.

Show code cell source

Hide code cell source

 
messages = [
    {
        "role": "system",
        "content": "You are a data extraction assistant. Respond only with JSON.",
    },
    {
        "role": "user",
        "content": (
            "Extract fields {name, organization, role, location} as JSON.\n\n"
            "Example:\n"
            "Input: John Smith, CEO of TechCorp in New York, oversees global strategy.\n"
            "Output: {\"name\": \"John Smith\", \"organization\": \"TechCorp\", "
            "\"role\": \"CEO\", \"location\": \"New York\"}\n\n"
            "Now extract from:\n"
            "Dr. Sarah Chen, a professor at MIT's CSAIL lab in Cambridge, "
            "leads research on neural machine translation."
        ),
    },
]

The single example locks in three things:

  1. Schema — exactly which keys appear in the JSON

  2. Style — title included in name, concise role description

  3. Format — flat JSON object, no nested structures

One-shot is the most cost-effective ICL strategy: a single well-chosen example captures the essential pattern while adding minimal context window overhead.

6.3.3.5. Few-Shot Prompting#

Few-shot prompting provides multiple examples, allowing the model to learn more complex patterns — handling edge cases, multiple entity types, or ambiguous inputs.

Show code cell source

Hide code cell source

 
few_shot_examples = """
Example 1:
Input: "The weather was terrible, and the service was slow."
Sentiment: negative

Example 2:
Input: "I loved the pasta — perfectly al dente and flavorful!"
Sentiment: positive

Example 3:
Input: "The room was okay. Nothing special, but nothing bad either."
Sentiment: neutral
"""

messages = [
    {
        "role": "system",
        "content": "You classify customer reviews. Respond with only one word: positive, negative, or neutral.",
    },
    {
        "role": "user",
        "content": (
            f"{few_shot_examples}\n"
            "Now classify:\n"
            "Input: \"The product works as advertised, though setup took longer than expected.\"\n"
            "Sentiment:"
        ),
    },
]

6.3.3.5.1. Designing Effective Few-Shot Examples#

The quality of few-shot examples matters more than their quantity:

Design principle

Explanation

Diversity

Cover different input types — don’t show only easy or prototypical examples

Representativeness

Examples should reflect the distribution of real inputs

Consistent format

Every example must use the exact same format the model should produce

Edge cases

Include at least one example with an unusual input to prevent over-generalization

Order

Later examples have stronger influence (recency); put the most representative example last. Few-shot accuracy has been shown to vary substantially with example ordering alone [Lu et al., 2022]

6.3.3.5.2. How Many Examples?#

0 examples (zero-shot):   Fast, cheap; works for common tasks with clear format
1 example (one-shot):     Locks in format; best for structured output tasks
2–3 examples (few-shot):  Covers variation; recommended for classification and extraction
4–8 examples (many-shot): Diminishing returns; most benefit realized by 3–5 examples
8+ examples:              Context window cost rises; prefer fine-tuning instead

A practical rule: start with zero-shot, observe failure modes, add one example to address the most common failure, then add more only if needed.

The decision tree:

  1. Does the task have a well-known format (Q&A, summarization)? → Zero-shot

  2. Does the output have a specific schema (JSON fields, classification labels)? → One-shot

  3. Are outputs inconsistent across runs or edge cases problematic? → Few-shot

  4. Are failures still frequent after 5 examples? → Fine-tuning

6.3.3.7. Exercises and discussion#

  1. Shot comparison Choose a named entity recognition task and implement it with zero-shot, one-shot, and three-shot prompts. Run each on ten sentences and count schema errors (missing fields, wrong types). How much does each additional shot reduce errors?

  2. Example order effect Take a three-shot classification prompt and test three orderings of the examples: (a) easy-easy-hard, (b) hard-hard-easy, (c) balanced. Measure accuracy on 20 test inputs. Does example order affect performance?

  3. Quality vs. quantity Compare: (a) three high-quality diverse examples vs. (b) six mediocre similar examples for a sentiment classification task. Which achieves higher accuracy? What does this suggest about the diminishing returns of additional examples?
