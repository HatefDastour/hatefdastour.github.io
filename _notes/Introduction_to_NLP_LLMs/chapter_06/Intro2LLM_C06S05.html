6.3.5. Output Verification and Structured Generation#

6.3.5.1. Learning objectives#

By the end of this section, you should be able to:

  • Explain why zero-shot JSON generation is unreliable and what failure modes it exhibits.

  • Use one-shot prompting to lock in a JSON schema and improve output consistency.

  • Explain what grammar-constrained generation is and how it guarantees syntactically valid output.

  • Implement grammar-constrained JSON generation using llama-cpp-python with a BNF grammar.

  • Select the appropriate output verification strategy based on reliability requirements and available tools.

6.3.5.2. The Structured Output Problem#

When LLMs must produce structured output — JSON, YAML, CSV — that will be parsed programmatically, reliability becomes critical. A single malformed bracket, missing field, or extra prose line breaks the downstream parser.

Three failure modes are common:

Failure mode 1: Extra prose
  Expected: {"name": "Alice", "age": 30}
  Received: "Here is the JSON you requested:
             {"name": "Alice", "age": 30}
             Let me know if you need anything else!"

Failure mode 2: Schema drift
  Expected: {"name": "Alice", "age": 30}
  Received: {"full_name": "Alice Smith", "years": "30", "city": "Boston"}

Failure mode 3: Syntax error
  Expected: {"name": "Alice", "age": 30}
  Received: {"name": "Alice", "age": 30,}  ← trailing comma (invalid JSON)

Each failure mode requires a different mitigation strategy.

6.3.5.3. Zero-Shot JSON: Unreliable Schema#

The simplest approach instructs the model to produce JSON without an example:

Show code cell source

Hide code cell source

 
messages = [
    {
        "role": "system",
        "content": "You are a data extraction assistant. Respond only with valid JSON.",
    },
    {
        "role": "user",
        "content": (
            "Extract the following fields from the text and return them as a JSON object: "
            "{name, organization, role, location}. Return ONLY the JSON, no explanation.\n\n"
            "Text: Dr. Sarah Chen, a professor at MIT's CSAIL lab in Cambridge, "
            "leads research on neural machine translation."
        ),
    },
]
output = pipe(messages, do_sample=False, temperature=None)[0]["generated_text"]
print(output)

 ```json

{

  "name": "Dr. Sarah Chen",

  "organization": "MIT's CSAIL lab",

  "role": "professor",

  "location": "Cambridge"

}

```

Zero-shot JSON works when the model is familiar with the schema. On novel or unusual field names, it frequently drifts — choosing different key names, adding unrequested fields, or including prose.

Mitigation for extra prose: post-process the output to extract JSON using a regex or json.loads within a try/except, then re-prompt if parsing fails.

6.3.5.4. One-Shot JSON: Schema-Locked Output#

Providing one example that demonstrates the exact expected schema dramatically reduces schema drift:

Show code cell source

Hide code cell source

 
messages = [
    {
        "role": "system",
        "content": "You are a data extraction assistant. Respond only with valid JSON.",
    },
    {
        "role": "user",
        "content": (
            "Extract fields {name, organization, role, location} as a JSON object.\n\n"
            "Example input: John Smith, CEO of TechCorp in New York, oversees global strategy.\n"
            'Example output: {"name": "John Smith", "organization": "TechCorp", '
            '"role": "CEO", "location": "New York"}\n\n'
            "Now extract from:\n"
            "Dr. Sarah Chen, a professor at MIT's CSAIL lab in Cambridge, "
            "leads research on neural machine translation.\n"
            "Return ONLY the JSON."
        ),
    },
]
output = pipe(messages, do_sample=False, temperature=None)[0]["generated_text"]
print(output)

 ```json

{

  "name": "Dr. Sarah Chen",

  "organization": "MIT's CSAIL lab",

  "role": "professor",

  "location": "Cambridge"

}

```

One-shot JSON substantially reduces schema drift and typically eliminates extra-prose failures because the model mirrors the example format. Trailing-comma and other syntax errors can still occur occasionally.

6.3.5.5. Output Verification Loop#

Even with one-shot prompting, validation is essential in production systems. A robust output loop catches failures and re-prompts:

Show code cell source

Hide code cell source

 
import json

def extract_with_retry(text: str, max_retries: int = 3) -> dict:
    schema_example = '{"name": "...", "organization": "...", "role": "...", "location": "..."}'

    for attempt in range(max_retries):
        messages = [
            {
                "role": "system",
                "content": "Respond only with valid JSON. No prose, no markdown fences.",
            },
            {
                "role": "user",
                "content": (
                    f"Extract {{name, organization, role, location}} from:\n{text}\n\n"
                    f"Use this exact schema: {schema_example}"
                ),
            },
        ]

        raw = pipe(messages, do_sample=False)[0]["generated_text"].strip()

        # Strip markdown code fences if present
        if raw.startswith("```"):
            raw = raw.split("```")[1]
            if raw.startswith("json"):
                raw = raw[4:]

        try:
            return json.loads(raw)
        except json.JSONDecodeError as e:
            if attempt < max_retries - 1:
                print(f"Parse failed (attempt {attempt + 1}): {e}")
            else:
                raise ValueError(f"Failed after {max_retries} attempts. Last output: {raw}")

6.3.5.6. Grammar-Constrained Generation: Guaranteed JSON#

The most reliable approach to structured generation is grammar-constrained decoding [Geng et al., 2023] — enforcing syntactic validity at the token level. Instead of post-hoc validation, the model is prevented from generating tokens that would violate the specified grammar.

llama-cpp-python, the Python binding for the llama.cpp inference engine [Gerganov and contributors, 2023], supports grammar-constrained generation via a BNF (Backus-Naur Form) grammar specification:

Show code cell source

Hide code cell source

 
from llama_cpp import Llama
import json

# Load Phi-3 in GGUF quantized format
llm = Llama.from_pretrained(
    repo_id="microsoft/Phi-3-mini-4k-instruct-gguf",
    filename="*fp16.gguf",
    n_gpu_layers=-1,   # Offload all layers to GPU
    n_ctx=2048,        # Context window (use half to save VRAM)
    verbose=False
)

print("GGUF model loaded via llama-cpp-python.")

# Define a JSON grammar that guarantees valid output structure
json_grammar_str = r"""
root   ::= object
value  ::= object | array | string | number | ("true" | "false" | "null") ws
object ::=
  "{" ws (
    string ":" ws value
    ("," ws string ":" ws value)*
  )? "}" ws
array  ::=
  "[" ws (value ("," ws value)*)? "]" ws
string ::= "\"" ([^"\\] | "\\" .)* "\"" ws
number ::= ("-"? ([0-9] | [1-9][0-9]*)) ("." [0-9]+)? ([eE] [-+]? [0-9]+)? ws
ws     ::= ([ \t\n] ws)?
"""

grammar = LlamaGrammar.from_string(json_grammar_str)

prompt = (
    "<|user|>\nExtract {name, organization, role, location} as JSON from:\n"
    "Dr. Sarah Chen, a professor at MIT's CSAIL lab in Cambridge, "
    "leads research on neural machine translation.<|end|>\n<|assistant|>\n"
)

output = llm(
    prompt,
    max_tokens=200,
    grammar=grammar,
    temperature=0.0,
)
print(output["choices"][0]["text"])

GGUF model loaded via llama-cpp-python.
{
  "name": "Dr. Sarah Chen",
  "organization": "MIT's CSAIL lab",
  "role": "professor leading research on neural machine translation",
  "location": "Cambridge"
}

With grammar-constrained decoding, the model cannot generate syntactically invalid JSON — every token it emits is constrained to be valid according to the grammar. The output is guaranteed to be parseable, eliminating both the extra-prose and syntax-error failure modes.

6.3.5.7. Output Verification Strategy Comparison#

Scenario

Recommended strategy

Prototype or low-stakes application

Zero-shot with try/except

Production application, standard schemas

One-shot + validation loop

Safety-critical or financial application

Grammar-constrained generation

Custom nested schemas

Grammar-constrained with custom BNF

Output fed directly to database or API

Grammar-constrained generation

The choice is ultimately a reliability-cost trade-off: grammar-constrained generation adds dependency on llama-cpp-python and requires a GGUF-format model, but eliminates an entire class of runtime failures that would otherwise require careful post-hoc handling.

6.3.5.8. Exercises and discussion#

  1. Failure mode inventory Run zero-shot JSON extraction on 20 different input sentences. Categorize each failure into: extra prose, schema drift, syntax error, or correct output. What is the failure rate, and which failure mode is most common?

  2. One-shot improvement Repeat the same 20 inputs with one-shot JSON prompting. How much does one example reduce each failure mode? Which failure mode is most resistant to one-shot mitigation?

  3. Grammar design Extend the BNF grammar above to enforce that the JSON object must contain exactly the keys name, organization, role, and location (no additional keys allowed, no missing keys). Test your grammar on inputs where the model would normally add extra fields. Does the grammar successfully prevent extra-field generation?
