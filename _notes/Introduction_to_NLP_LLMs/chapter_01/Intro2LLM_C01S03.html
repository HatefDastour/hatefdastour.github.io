1.3. Text Preprocessing and Normalization#

1.3.1. Learning Objectives#

By the end of this section, you should be able to:

  • Explain why text preprocessing and normalization are needed in NLP workflows.

  • Choose an appropriate normalization intensity level (light, moderate, or aggressive) for a given task.

  • Implement common preprocessing operations in Python using both NLTK and spaCy (cleaning, tokenization, sentence segmentation).

  • Compare NLTK tokenization strategies and select the right one for your text type.

  • Distinguish between stemming and lemmatization and understand when each is appropriate.

  • Remove stopwords thoughtfully, understanding when their removal can hurt performance.

  • Describe the role of subword tokenization in modern language models and explain Byte-Pair Encoding at a high level.

  • Discuss situations where aggressive normalization is harmful, especially with LLM prompts.

Section Goal

Raw text is rarely usable as-is. This section equips you with a practical toolkit for turning messy text into clean, structured input for downstream models. You will learn how to balance simplicity with preservation of meaning, compare classical approaches (stemming, stopword removal) with modern ones (subword tokenization), and develop an intuition for when preprocessing helps and when it hurts.

1.3.2. Why Preprocessing?#

Raw text is messy. Before we can train models or call an LLM API, we almost always need to clean and normalize our input. Reasons include:

  • Removing artifacts of data collection (HTML tags, boilerplate, markup).

  • Standardizing variations that do not carry meaning for the task (case, punctuation, spacing).

  • Preparing data for specific models or toolkits (e.g., tokenization requirements).

At the same time, over-preprocessing can destroy signal. A key theme of this section is learning how to balance simplicity and preservation of meaning.

In applied settings — such as extracting information from environmental impact reports or triaging public-health bulletins — the preprocessing decisions you make can materially change the conclusions you draw from the data. It is therefore important to treat preprocessing as part of the modeling design, not just a mechanical first step.

1.3.3. The Preprocessing Pipeline at a Glance#

Understanding how each step fits into the broader pipeline helps you make informed decisions about what to apply and in what order.

1.3.4. Cleaning and Normalization#

The goal of cleaning and normalization is to reduce superficial variation that does not carry meaning for our task. Typical operations include:

  • Lowercasing.

  • Removing or standardizing punctuation.

  • Handling digits, URLs, emojis, and hashtags.

  • Collapsing repeated whitespace.

  • Fixing character encoding issues.

1.3.4.1. A Primer on Regular Expressions#

Most text cleaning in Python relies on regular expressions (regex), a mini-language for describing and matching text patterns. The re module is part of the Python standard library.

Key Term — Regular Expression

A regular expression is a sequence of characters that defines a search pattern. Python’s re module lets you search for, replace, and split text using such patterns.

Common issues and fixes:

  • UTF-8 is the standard encoding for most modern web content.

  • Latin-1 (ISO-8859-1) is common in older European documents. If you see garbled characters like â€™ instead of ', you are likely reading UTF-8 bytes as Latin-1.

  • Unicode normalization collapses visually identical characters that have different code points. For example, é can be encoded as a single character (U+00E9) or as e followed by a combining accent (U+0065 U+0301). Use unicodedata.normalize("NFC", text) to canonicalize.

normalized = unicodedata.normalize("NFC", text)
print(repr(text))        # 'cafe\u0301'
print(repr(normalized))  # 'caf\xe9'   ← single character é

1.3.4.3. Normalization Levels#

Different tasks call for different levels of aggressiveness. A practical framework defines three levels:

Show code cell source

Hide code cell source

 
import re
import string

def normalize_text(text: str, level: str = "moderate") -> str:
    """
    Normalize text at three intensity levels.

    Args:
        text  (str): Input text to normalize.
        level (str): 'light', 'moderate', or 'aggressive'.

    Returns:
        str: Normalized text.
    """
    if level == "light":
        # Only strip leading/trailing whitespace
        return text.strip()

    elif level == "moderate":
        text = text.lower()
        text = re.sub(r"\d+", "", text)         # remove digits
        text = text.translate(                   # remove punctuation
            str.maketrans("", "", string.punctuation)
        )
        return " ".join(text.split())            # collapse whitespace

    elif level == "aggressive":
        text = text.lower()
        text = re.sub(r"[^a-zA-Z\s]", "", text) # keep letters and spaces only
        return re.sub(r"\s+", " ", text).strip()

    else:
        raise ValueError("Level must be 'light', 'moderate', or 'aggressive'")

Here string.punctuation is a convenience constant from Python’s standard library containing all ASCII punctuation characters (!"#$%&'()*+,-./:;<=>?@[\]^_{|}~``). Using str.maketrans` with it builds a fast character-deletion mapping.

Running this across three levels on a sample sentence:

What to notice:

  • Light normalization preserves numbers, punctuation, and capitalization — important for NER and numerical reasoning.

  • Moderate normalization produces clean, lowercase word tokens — suitable for topic modeling and general classification.

  • Aggressive normalization strips everything non-alphabetic — useful for pure bag-of-words analysis but loses numerical and punctuation signals.

1.3.4.4. Domain-Aware Normalization#

Different domains require different choices:

  • Scientific text: preserve equations, citations, and special notation.

  • Clinical notes: abbreviations and shorthand might be crucial; aggressive normalization can erase important cues.

  • Environmental and policy reports: numeric values, units, and geographic names are often critical.

  • Social media: emojis, hashtags, and creative spellings often encode sentiment and stance.

When designing preprocessing pipelines, always ask: What information might my model need to solve the task?

1.3.4.5. Handling Social Media Text#

Social media text (tweets, posts, forum comments) requires special consideration. Emojis, hashtags, user mentions, and creative spelling all carry meaning that standard preprocessing may erase:

Show code cell source

Hide code cell source

 
import re

def preprocess_social(text: str) -> str:
    """Light-touch preprocessing designed for social media text."""
    text = re.sub(r"https?://\S+|www\.\S+", "<URL>", text)   # URLs
    text = re.sub(r"@\w+", "<USER>", text)                    # mentions
    text = re.sub(r"#(\w+)", r"\1", text)                     # hashtags → plain words
    text = re.sub(r"(.)\1{2,}", r"\1\1", text)                # sooooo → soo
    text = re.sub(r"\s+", " ", text).strip()
    return text


tweets = [
    "@EPA please check this: https://floodreport.org #flooding #ClimateChange",
    "OMG the river is sooooo high!!! #flood #scared",
    "New study shows CO2 levels are wayyyy too high 😱 #science",
]

for tweet in tweets:
    print(f"Original : {tweet}")
    print(f"Processed: {preprocess_social(tweet)}")
    print()

Note on Emojis

Emojis encode sentiment and stance information. Rather than stripping them, consider either converting them to their textual description using the emoji library:

import emoji
emoji.demojize("😱")   # → ":face_screaming_in_fear:"

or using a model that was fine-tuned on social media text and handles emojis natively.

1.3.5. Tokenization and Sentence Segmentation#

Tokenization splits text into meaningful units called tokens. Sentence segmentation (or sentence boundary detection) divides a document into individual sentences. While simple str.split() works for quick experiments, robust tokenization is essential for production systems.

Key Term — Token

A token is the atomic unit of text that a model processes. For classical NLP, tokens are words (or punctuation marks treated separately). For modern LLMs, tokens are subword pieces produced by algorithms like BPE. The choice of what counts as a token is a fundamental modeling decision.

NLTK provides several tokenization strategies. Understanding their differences lets you choose the right one:

import nltk
from nltk.tokenize import (
    word_tokenize, sent_tokenize,
    regexp_tokenize, wordpunct_tokenize
)

nltk.download("punkt", quiet=True)
nltk.download("punkt_tab", quiet=True)

text = "Transformers changed NLP. LLMs build on transformers — they're not just bigger RNNs!"

print("word_tokenize     :", word_tokenize(text)[:12])
print("wordpunct_tokenize:", wordpunct_tokenize(text)[:12])
print("regexp (\\w+)      :", regexp_tokenize(text, r"\w+")[:12])
print("sent_tokenize     :", sent_tokenize(text))

word_tokenize     : ['Transformers', 'changed', 'NLP', '.', 'LLMs', 'build', 'on', 'transformers', '—', 'they', "'re", 'not']
wordpunct_tokenize: ['Transformers', 'changed', 'NLP', '.', 'LLMs', 'build', 'on', 'transformers', '—', 'they', "'", 're']
regexp (\w+)      : ['Transformers', 'changed', 'NLP', 'LLMs', 'build', 'on', 'transformers', 'they', 're', 'not', 'just', 'bigger']
sent_tokenize     : ['Transformers changed NLP.', "LLMs build on transformers — they're not just bigger RNNs!"]

Notice that word_tokenize correctly separates the contraction they're into ["they", "'re"], preserving both the subject and the auxiliary verb as distinct tokens. regexp_tokenize(r"\w+") discards all punctuation, including the apostrophe — useful when you need a clean word list but not when contractions carry meaning.

1.3.5.2. Tokenization Edge Cases#

Real text contains many challenging patterns:

  • Abbreviations: "Dr.", "U.S.A.", "e.g." — the period should not trigger sentence splitting.

  • Contractions: "can't", "it's", "they've" — splitting on apostrophes should produce valid tokens.

  • Hyphenated words: "state-of-the-art", "CO₂-driven" — should these be one token or three?

  • Multi-word expressions: "New York", "ice cream" — sometimes a phrase is semantically one unit.

NLTK’s word_tokenize uses a trained model (PunktTokenizer) to resolve many of these cases, but it may still fail on domain-specific text (e.g., clinical abbreviations). When working in specialized domains, consider custom tokenization rules.

1.3.6. Stopwords, Stemming, and Lemmatization#

Many NLP tasks benefit from reducing inflected forms and removing very common function words. This section connects directly to the tokenization and normalization lab.

1.3.6.1. Stopword Removal#

Stopwords are very common words (e.g., “the”, “is”, “and”, “in”) that carry little discriminative information for many tasks. NLTK provides a built-in stopword list:

Show code cell source

Hide code cell source

 
import nltk
from nltk.corpus import stopwords

nltk.download("stopwords", quiet=True)

stops = set(stopwords.words("english"))
tokens = ["Running", "large", "language", "models", "are", "the", "future", "of", "NLP"]

filtered = [t for t in tokens if t.lower() not in stops]
print("Original :", tokens)
print("Filtered :", filtered)

Original : ['Running', 'large', 'language', 'models', 'are', 'the', 'future', 'of', 'NLP']
Filtered : ['Running', 'large', 'language', 'models', 'future', 'NLP']

spaCy marks each token with token.is_stop so you can filter programmatically without maintaining a separate word list:

Show code cell source

Hide code cell source

 
nlp = spacy.load("en_core_web_sm")
doc = nlp("Large language models are the future of NLP.")
content_tokens = [t.text for t in doc if not t.is_stop and not t.is_punct]
print("Content tokens:", content_tokens)

Content tokens: ['Large', 'language', 'models', 'future', 'NLP']

Warning: Whether you should remove stopwords depends strongly on the task. Removing them is common for topic modeling and document retrieval; keeping them is often better for tasks that depend on function words (e.g., question answering, entailment, and many classification tasks). The phrase “not bad” changes meaning entirely if “not” is removed as a stopword.

1.3.6.2. Stemming#

Stemming applies heuristic rules to strip affixes and reduce words to an approximate root form, called a stem. It does not always produce real words — its goal is to map inflected forms to the same base so they can be counted together.

Show code cell source

Hide code cell source

 
from nltk.stem import PorterStemmer, SnowballStemmer, LancasterStemmer

    (SnowballStemmer("english"), "Snowball"),

1.3.6.3. Lemmatization#

Lemmatization uses vocabulary and morphological analysis to map words to their canonical dictionary form, called a lemma. Unlike stemming, it always produces a real word.

NLTK’s WordNetLemmatizer requires explicit part-of-speech information to produce the correct lemma:

Show code cell source

Hide code cell source

 
from nltk.stem import WordNetLemmatizer
import nltk

nltk.download("wordnet", quiet=True)
lemmatizer = WordNetLemmatizer()

verbs = ["running", "runs", "ran", "better"]
nouns = ["wolves", "mice", "geese", "children"]

# For NLTK, the POS must be specified: 'v' = verb, 'n' = noun
print("Verb lemmas:", [lemmatizer.lemmatize(w, pos="v") for w in verbs])
print("Noun lemmas:", [lemmatizer.lemmatize(w, pos="n") for w in nouns])

Verb lemmas: ['run', 'run', 'run', 'better']
Noun lemmas: ['wolf', 'mouse', 'goose', 'child']

Without the correct POS tag, WordNetLemmatizer defaults to treating input as a noun, which will incorrectly lemmatize verbs (e.g., "ran" → "ran" instead of "run"). This is a common source of errors when using NLTK lemmatization.

spaCy handles POS-aware lemmatization automatically — no manual POS specification is needed:

Show code cell source

Hide code cell source

 
import spacy

nlp = spacy.load("en_core_web_sm")
tokens = ["running", "runs", "ran", "better", "wolves", "mice"]
doc = nlp(" ".join(tokens))

for t in doc:
    print(f"{t.text:<12} → lemma: {t.lemma_:<12} (POS: {t.pos_})")

running      → lemma: run          (POS: VERB)
runs         → lemma: run          (POS: NOUN)
ran          → lemma: run          (POS: VERB)
better       → lemma: well         (POS: ADJ)

1.3.6.4. Comparing Stemming and Lemmatization#

from nltk.stem import PorterStemmer, WordNetLemmatizer
from nltk.corpus import stopwords

stops    = set(stopwords.words("english"))
stemmer  = PorterStemmer()
lemmatizer = WordNetLemmatizer()

tokens   = ["running", "runs", "ran", "runner", "the", "and"]
filtered = [t for t in tokens if t.lower() not in stops]
stems    = [stemmer.stem(t)            for t in filtered]
lemmas   = [lemmatizer.lemmatize(t, pos="v") for t in filtered]

print("Filtered:", filtered)
print("Stems   :", stems)
print("Lemmas  :", lemmas)

Filtered: ['running', 'runs', 'ran', 'runner']
Stems   : ['run', 'run', 'ran', 'runner']
Lemmas  : ['run', 'run', 'run', 'runner']

Summary of differences:

  • Stemming is faster and requires no lexical resource, but may produce non-standard forms (e.g., "runner" → "runner" with Porter, "runn" with Lancaster).

  • Lemmatization is slower but produces real dictionary words, making outputs more interpretable. Prefer lemmatization when outputs are shown to end users or when interpretability matters.

1.3.7. Subword Tokenization and LLMs#

Modern LLMs rarely operate on whitespace-delimited words. Instead, they use subword tokenization schemes that split text into pieces smaller than a word but larger than a single character.

1.3.7.1. Why Subword Tokenization?#

Two problems motivate subword tokenization:

  1. Out-of-vocabulary (OOV) words: A word-level vocabulary cannot represent words not seen during training (new technical terms, proper names, misspellings).

  2. Vocabulary explosion: Morphologically rich languages (Finnish, Turkish, German) produce huge vocabularies when every inflected form is a separate word.

Subword tokenization solves both by decomposing rare or unknown words into frequent, smaller pieces.

1.3.7.2. Byte-Pair Encoding (BPE)#

The most widely used subword algorithm is Byte-Pair Encoding (BPE), originally developed for data compression and adapted for NLP.

How BPE works (conceptually):

  1. Start with a vocabulary of individual characters (a, b, c, …, z, 0–9, punctuation, …).

  2. Count all pairs of adjacent symbols in the training corpus.

  3. Merge the most frequent pair into a new symbol (e.g., merge e + s → es).

  4. Repeat steps 2–3 for a fixed number of merge operations (typically 30,000–50,000).

  5. Apply the learned merge rules to tokenize new text.

For example, after training on a large corpus, BPE might tokenize:

"internationalization"  →  ["international", "ization"]
"tokenizing"            →  ["token", "izing"]
"unhappiness"           →  ["un", "happiness"]

Common subwords (like "tion", "ing", "un") are reused across many words, dramatically reducing vocabulary size.

WordPiece (used by BERT) works similarly but uses a different merge criterion (maximizing likelihood rather than frequency). Subwords that are not word-initial are prefixed with ##:

"tokenizing"  →  ["token", "##izing"]

1.3.7.3. Subword Tokenization in Practice#

from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("distilbert-base-uncased")

text = "Lemmatization reduces words to their canonical forms."

Subword tokens: ['[CLS]', 'le', '##mma', '##ti', '##zation', 'reduces', 'words', 'to', 'their', 'canonical', 'forms', '.', '[SEP]']
Token count: 13

From a practical standpoint:

  • When using APIs or libraries for LLMs, you typically do not need to tokenize manually; the library handles it.

  • When training or fine-tuning models, you must use the same tokenizer as the base model — mixing tokenizers will corrupt the input.

  • Context length limits (e.g., 4,096 or 128,000 tokens) are measured in subword tokens, not words. A 1,000-word document typically produces 1,200–1,500 subword tokens.

  • Understanding subword boundaries can help with prompt engineering: short, common words are usually single tokens, while rare technical terms may be split into many pieces.

1.3.8. When Normalization Can Hurt#

Although normalization is powerful, overly aggressive cleaning can remove important signal:

  • Stripping punctuation may erase cues such as question marks (indicating interrogative intent) or exclamation points (indicating emphasis or urgency).

  • Lowercasing may obscure proper names ("NATO" vs. "nato") or language distinctions (German nouns are capitalized by convention).

  • Removing stopwords may harm tasks that depend on function words (e.g., question answering, logical inference, coreference resolution).

  • Replacing all digits with a generic <NUM> token might hurt numerical reasoning tasks (e.g., comparing measurements or dates).

  • Unicode normalization that strips diacritics may collapse distinct words in French, Spanish, or German ("côte" ≠ "cote").

Modern LLMs are often robust to raw, noisy text because their tokenizers and training corpora cover many variants. However, thoughtful preprocessing remains important when you:

  • Train your own models on limited data.

  • Work with specialized domains (clinical notes, legal documents, scientific articles, environmental reports).

  • Need interpretable features and transparent transformations.

  • Want to reduce input length while preserving task-relevant information.

Key Takeaways

  • Preprocessing transforms raw, messy text into a form that models can use. The right pipeline depends entirely on the task, domain, and downstream model.

  • The preprocessing pipeline flows through: cleaning → tokenization → normalization → noise removal → stopword removal → stemming/lemmatization.

  • Regular expressions (re module) are the workhorse of text cleaning; mastering a small set of patterns covers the vast majority of use cases.

  • Encoding matters: always read files with an explicit encoding and normalize Unicode before processing.

  • Normalization levels (light, moderate, aggressive) encode a deliberate tradeoff: aggressive normalization reduces vocabulary size but can destroy signal.

  • Tokenization is not trivial: word_tokenize, wordpunct_tokenize, and regex tokenizers each make different assumptions about contractions and punctuation. Real text contains abbreviations, hyphenated words, and contractions that challenge simple rules.

  • Stopword removal helps for topic modeling and term-frequency analysis, but can hurt tasks that rely on function words (question answering, entailment).

  • Stemming is fast and resource-free but may produce non-words. Lemmatization is slower but produces real dictionary forms — prefer it when interpretability matters or outputs are user-facing. With NLTK, always supply the correct POS to the lemmatizer.

  • Subword tokenization (BPE, WordPiece) is what LLMs actually use. Context-length limits are measured in subword tokens, not words. Always use the same tokenizer as the pretrained model you are working with.

  • When in doubt, prefer lighter normalization, especially for LLM-based pipelines where the model has already been trained on diverse text.
